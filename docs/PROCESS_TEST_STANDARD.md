# Hotel PMS 프로세스 테스트 기준

## 목적

이 문서는 Hotel PMS 수정 후 “테스트 완료”라고 판단하는 기준을 고정하기 위한 문서다.

전체 페이지별 실행 케이스와 순서는 [PMS 전체 통합 테스트 계획서](./PMS_FULL_INTEGRATION_TEST_PLAN_20260718.md)를 기준으로 한다. 이 기준 문서는 판정 원칙을, 상세 계획서는 현재 제품 페이지와 Case ID를 관리한다.

앞으로 테스트 완료는 단순히 스크립트가 통과했거나 버튼을 한 번 눌러본 상태가 아니라, 실제 페이지에서 사용자가 하는 순서대로 입력, 수정, 저장, 재조회, 다국어 전환, 화면 크기 확인까지 통과했을 때만 말한다.

## 테스트 판정 원칙

1. 실제 화면 기준으로 테스트한다.
   - 대상 URL의 실제 페이지를 열고 버튼, 입력창, 풀다운, 팝업을 사용자가 쓰는 방식으로 조작한다.
   - 내부 상태를 직접 바꾸거나 함수만 호출한 결과는 “로직 검증”이지 “유저 테스트 통과”로 보지 않는다.

2. 저장 기능은 반드시 재조회로 확인한다.
   - 등록, 수정, 삭제, 상태 변경은 저장 직후 화면 변화만 보지 않는다.
   - 팝업 재오픈, 페이지 새로고침, 다른 메뉴 이동 후 복귀 중 하나 이상으로 유지 여부를 확인한다.

3. 프로세스 단위로 통과해야 한다.
   - 예: 신규 예약은 객실 선택, 청소 상태 경고, 투숙객 등록, 저장, 카드 반영, 상세 팝업 재조회까지 한 케이스다.
   - 중간 버튼 하나만 성공해도 전체 프로세스가 실패하면 실패로 기록한다.

4. 다국어는 화면 전환 후 초기 렌더링까지 확인한다.
   - KR -> EN, EN -> KR 변경 직후만 보지 않는다.
   - 언어 변경 후 메뉴 이동, 팝업 오픈, 풀다운 오픈, 입력 placeholder, 브라우저 title까지 확인한다.
   - 한글이 잠깐 나왔다가 영어로 바뀌는 경우도 실패다.

5. 레이아웃은 실제 사용 크기별로 본다.
   - 데스크톱, 태블릿, 모바일 폭에서 버튼 넘침, 호버 깜빡임, 풀다운 흔들림, 카드 색상 불일치, 텍스트 잘림을 확인한다.

6. 테스트 증거를 남긴다.
   - 수정 단위마다 테스트한 케이스 ID, 결과, 실패 시 화면/콘솔/데이터 상태를 기록한다.
   - “전체 문제 없음”이라는 표현은 아래 필수 케이스가 실제로 통과한 경우에만 사용한다.

## 실패로 보는 기준

- 저장 후 새로고침 또는 재오픈 시 데이터가 원복된다.
- 목록에서 누른 객실/예약과 팝업의 객실/예약이 다르다.
- 상태 버튼을 눌렀는데 실제 상태, 버튼 라벨, 필터 카운트 중 하나라도 맞지 않는다.
- 다국어 모드에서 한글/영문이 섞인다.
- 버튼, 날짜 입력, 풀다운, 삭제, 수정 중 하나라도 클릭되지 않는다.
- 호버 시 버튼이나 카드가 깜빡이거나 레이아웃이 움직인다.
- 필터 카운트와 실제 목록 수가 다르다.
- console error가 발생한다.
- 모바일/태블릿에서 팝업, 풀다운, 리스트 클릭이 동작하지 않는다.

## 우선순위

### P0: 운영 차단

- 예약 등록, 체크인, 체크아웃, 객실 이동, 예약 취소
- 정산 완료, 정산 미완료 전환, 결제 등록
- 부가서비스 등록, 완료, 미완료 전환
- 직원 로그인/권한/상태
- 데이터 저장 후 재조회 불일치

### P1: 시연/운영 신뢰도

- 다국어 라벨, placeholder, 풀다운, 팝업, 브라우저 title
- 필터 카운트와 실제 목록 일치
- 모바일/태블릿 클릭, 풀다운 안정성
- 운영 감사 로그 기록
- 카드 색상, 상태 배지 일관성

### P2: 화면 품질

- 버튼 길이, 줄바꿈, 정렬
- KPI 카드 디자인
- 인쇄/미리보기 레이아웃
- 표 컬럼 너비와 합계 표시

## 공통 테스트 케이스 양식

| 항목 | 내용 |
| --- | --- |
| Case ID | 예: RES-BOARD-NEW-001 |
| 화면 | 예: 예약 현황 |
| 사전 데이터 | 테스트 전 필요한 객실, 예약, 상태 |
| 절차 | 실제 클릭/입력 순서 |
| 기대 결과 | 화면, 데이터, 로그, 다국어 기대값 |

| 판정 | Pass / Fail / Blocked |
| 증거 | 스크린샷, 콘솔, 저장 데이터, 커밋 |
| 비고 | 재현 조건, 브라우저, viewport |

## 테스트 결과 보고 기준

수정 리포트에는 다음만 적는다.

1. 수정한 파일
2. 고친 문제
3. 통과한 Case ID
4. 실패 또는 미확인 Case ID
5. 남은 리스크
6. 커밋 해시

“전체 테스트 완료”는 필수 P0/P1 케이스가 모두 Pass일 때만 사용한다.

## 앞으로의 적용 방식

- 버그를 수정하면 해당 화면의 Case ID를 반드시 실행한다.
- 새 버그가 나오면 이 문서에 Case ID를 추가하고 같은 종류의 문제를 다른 화면에서도 찾는다.
- 자동 테스트는 보조 수단이다. 최종 판정은 실제 페이지 기반 유저 플로우 테스트로 한다.
- 사용자가 제보한 화면과 내 테스트 화면이 다르면, 내 테스트는 실패로 보고 제보 화면 조건을 테스트 케이스에 반영한다.
