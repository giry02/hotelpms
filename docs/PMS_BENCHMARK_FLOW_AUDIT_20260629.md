# Hotel PMS 흐름 테스트 및 벤치마킹 리포트

작성일: 2026-06-29  
대상: Hotel PMS GitHub Pages/로컬 정적 화면, mock API 데이터, 주요 운영 흐름  
관점: 호텔 PMS 운영자 관점, 시스템 관점, 투숙객/단체 대표 관점

## 1. 결론 요약

1.1. 현재 시스템은 예약, 단체, 객실, 하우스키핑, 정산, CRM, 감사 로그까지 화면 범위는 넓고 자동 스모크 기준으로는 안정적입니다.

1.2. 다만 호텔 실무 흐름 기준으로는 `예약 생성 -> 객실 배정 -> 체크인 -> 재실 관리 -> 부가서비스 -> 체크아웃 -> 정산 -> 마감 -> 감사/리포트`가 한 줄로 이어진다는 느낌이 아직 약합니다.

1.3. 가장 큰 개선 포인트는 새 기능 추가보다 "상태 기준 통일", "다음 업무로 넘어가는 CTA", "업무 마감 전 미처리 항목의 자동 큐", "동일 데이터의 단일 원천화"입니다.

1.4. 벤치마킹 PMS들은 대부분 예약/객실/결제/하우스키핑을 분리 메뉴로만 두지 않고, 오늘 해야 할 업무 큐와 객실 단위 상태 전이를 중심으로 운영합니다.

1.5. 본 시스템은 시연 데이터와 화면 수는 충분하지만, 업무자가 판단해야 하는 "왜 이 상태인지", "다음에 무엇을 해야 하는지", "어디에서 처리해야 하는지"를 더 명확히 해야 합니다.

## 3. 벤치마킹 기준

3.1. BM-01. Cloudbeds PMS

- 참고: [Cloudbeds Daily Operations Guide](https://myfrontdesk.cloudbeds.com/hc/en-us/articles/40771665486363-Your-Cloudbeds-PMS-Daily-Operations-Guide)
- 관찰: 일일 운영을 도착/재실/출발, 객실 상태, 결제/정산, 리포트처럼 반복 업무 단위로 묶어 안내합니다.
- 시사점: 현재 시스템도 화면은 나뉘어 있으나, "오늘 해야 할 일" 큐가 각 화면에 흩어져 있어 대시보드와 일일 마감이 중간 허브 역할을 더 해야 합니다.

3.2. BM-02. Mews

- 참고: [Mews Hospitality Cloud](https://www.mews.com/en/products/property-management-system)
- 관찰: PMS를 예약 관리만이 아니라 운영 자동화, 결제, 게스트 여정, 하우스키핑까지 연결하는 운영 플랫폼으로 설명합니다.
- 시사점: 본 시스템은 부가서비스/CRM/정산까지 범위가 넓으므로, 모듈 간 데이터 이동을 "링크"가 아니라 "업무 전이"로 보여줘야 합니다.

3.3. BM-03. Oracle Hospitality OPERA Cloud

- 참고: [Oracle Hospitality OPERA Cloud Property Management](https://www.oracle.com/hospitality/hotel-property-management/)
- 관찰: 대형 PMS는 예약, 프런트 오피스, 하우스키핑, 객실, 수납/캐셔링, 리포팅을 호텔 운영 표준 업무로 분리하면서도 같은 프로필/예약/folio 데이터 위에서 움직입니다.
- 시사점: 현재 mock API는 많이 정리되었지만 일부 화면은 여전히 화면별 localStorage/정적 데이터 흔적이 남아 있어 "데이터 단일 원천" 설계가 중요합니다.

3.4. BM-04. Little Hotelier / SiteMinder 계열 소형 숙소 PMS

- 참고: [Little Hotelier Property Management System](https://www.littlehotelier.com/property-management-system/)
- 관찰: 소형 숙소용 PMS도 예약 캘린더, 객실 배정, 하우스키핑, 게스트 커뮤니케이션, 채널/직접 예약을 한 플랫폼에서 연결하는 것을 핵심 가치로 둡니다.
- 시사점: 본 시스템은 채널/온라인 예약 쪽은 약하고 호텔 내부 운영 흐름은 강하므로, 내부 운영 데모 제품으로 갈지 채널 연동형 PMS로 갈지 제품 포지션을 정해야 합니다.

## 4. 현재 시스템의 강점

4.1. STR-01. 업무 모듈 폭이 넓습니다.

예약 현황, 예약 목록, 타임라인, 단체, 하우스키핑, 시설 보수, 부가서비스, 정산, CRM, 감사 로그까지 호텔 운영 데모에 필요한 핵심 화면이 대부분 존재합니다.

4.2. STR-02. 최근 사용자 지적 사항이 회귀 테스트로 축적되고 있습니다.

날짜 기준, 단체명 표시, 청소 상태 3종, 정산 기준, 일일 마감 시재 등 반복 이슈가 `USER_ACCEPTANCE_TEST_GUIDE.md`에 누적되어 있습니다.

4.3. STR-03. 자동 테스트 기반이 이미 쓸만합니다.

58개 페이지 스모크, 버튼 클릭, E2E 업무 흐름, 예약/단체 회귀 테스트가 통과합니다. 이는 리팩터링 전에 매우 좋은 안전망입니다.

4.4. STR-04. 단체/행사 데이터가 시연 가능한 수준으로 개선되어 있습니다.

회사 단체와 일반인 대표 단체가 모두 보이고, 단체 등록 관리와 행사 상세이 연결됩니다.

4.5. STR-05. 정산/마감의 호텔 업무 논리가 깊어지고 있습니다.

시재 인수인계, 현금/카드 분리, 중간출금, 전일 마감현금, 차액 여부처럼 실제 프런트 마감에서 중요한 개념이 반영되어 있습니다.

## 5. 호텔 PMS 운영자 관점 개선사항

5.1. OP-01. 대시보드는 "홍보/이벤트"보다 "오늘의 처리 큐"가 먼저 보여야 합니다.

- 현재 관찰: 대시보드 상단에 제휴 할인 이벤트 텍스트가 길게 노출되어, 체크인/체크아웃/미수금/청소/정산 같은 당일 업무보다 시선 우선순위가 높아 보입니다.
- 문제: 프런트 직원은 출근 직후 오늘 도착, 미도착, 출발, 미수금, DND, 점검 객실을 먼저 봐야 합니다.
- 개선: 대시보드 첫 화면을 `오늘 처리 큐` 중심으로 재구성합니다.
- 권장 구성:
  - 오늘 체크인 대기
  - 체크아웃 미처리
  - 미수금/예치금 확인
  - 청소 필요/DND
  - 점검 중/객실 이동 필요
  - 단체 도착/퇴실

5.2. OP-02. 예약 현황, 예약 목록, 타임라인의 상태명이 완전히 같아야 합니다.

- 현재 관찰: 예약 현황은 `투숙 중`, 예약 목록 KPI는 `재실 중`, 타임라인은 `점유 중/체크인 완료` 같은 표현이 섞입니다.
- 문제: 운영자는 같은 예약을 세 화면에서 보면서 서로 다른 상태처럼 오해할 수 있습니다.
- 개선: 공통 상태 사전을 만듭니다.
- 추천 상태:
  - `예약 확정`
  - `체크인 예정`
  - `투숙 중`
  - `체크아웃 예정`
  - `레이트 체크아웃`
  - `체크아웃 완료`
  - `취소`
  - `노쇼`
  - `단체 배정`
  - `점검 중`

5.3. OP-03. 예약 카드에서 다음 액션이 더 명확해야 합니다.

- 현재 관찰: 카드에는 상태, 청소, 금액, 버튼이 같이 보이지만 "지금 해야 할 액션"이 강하게 드러나지 않습니다.
- 문제: 체크인 예정 객실과 예약 없음 객실이 카드 구조가 비슷하면 숙련자가 아닌 직원은 판단 시간이 늘어납니다.
- 개선: 카드 하단에 상태별 주 CTA를 하나만 강조합니다.
- 예시:
  - 체크인 예정: `체크인 처리`
  - 투숙 중: `투숙 상세`
  - 체크아웃 예정: `정산/체크아웃`
  - 예약 없음: `예약 등록`
  - 청소 필요: `청소 배정`
  - 점검 중: `보수 티켓`

5.4. OP-04. 청소 상태와 숙박 상태의 위치를 고정해야 합니다.

- 현재 관찰: 최근 개선으로 청소 상태는 많이 단순화되었지만, 예약 카드와 목록에서 청소 선택지가 상태 정보와 같은 밀도로 보입니다.
- 문제: `청소 필요`가 객실 판매 가능 여부인지, 투숙 중 서비스 청소인지, 체크아웃 후 청소인지 즉시 구분하기 어렵습니다.
- 개선: 카드 주색은 숙박/점유 상태, 하단 작은 배지는 청소 상태로 고정합니다.
- 추가 제안: 청소 상태 풀다운에는 `방해금지`, `청소필요`, `청소완료` 3개만 유지하고, 상태 변경 시 변경 일시/담당자를 남깁니다.

5.5. OP-05. 단체 업무는 "행사 상세" 안에서 완료 가능해야 합니다.

- 현재 관찰: 행사 상세에는 개요, 객실 배정, 투숙객 명단, 정산, 변경 이력이 있어 방향은 좋습니다.
- 문제: 업무자가 단체 객실을 배정한 뒤 예약 현황/정산/명단 화면을 오가야 할 여지가 있습니다.
- 개선: 행사 상세를 단체 업무의 cockpit으로 사용합니다.
- 필요한 CTA:
  - 미배정 객실 `객실 배정 완료`
  - 미등록 대표 `대표 등록`
  - 체크인 당일 `단체 체크인`
  - 퇴실 당일 `단체 정산`
  - 변경 발생 `변경 이력 남기기`

5.6. OP-06. 정산 목록과 일일 마감 사이에 "미처리 정산 큐"가 필요합니다.

- 현재 관찰: 일일 마감에는 미처리 체크아웃, 미도착 체크인, 미결제 주문이 표시됩니다.
- 문제: 항목을 클릭했을 때 어떤 화면에서 어떤 행을 처리해야 하는지가 더 직접적이어야 합니다.
- 개선: 마감 체크의 각 미처리 항목에서 필터가 적용된 대상 목록으로 이동하거나, 마감 화면 안에서 drawer로 처리합니다.

5.7. OP-07. 전일 마감현금과 시작시재의 관계를 더 운영 언어로 설명해야 합니다.

- 현재 관찰: 시재 인수인계 구조는 좋아졌지만, 현금 흐름 공식은 초보자에게 여전히 어렵습니다.
- 개선: 마감 화면의 핵심 문구를 다음처럼 고정합니다.
  - `전일 마감현금`: 이전 근무자가 넘긴 실제 보유 현금
  - `시작시재`: 오늘 근무 시작 시 확인한 실제 보유 현금
  - `차액`: 전일 마감현금과 시작시재가 다를 때의 인수인계 차이
  - `기준 현금`: 시작시재 + 금일 현금 수납 + 예치금 - 중간출금

5.8. OP-08. 요금/등급/할인의 연결 흐름을 한 화면에서 설명해야 합니다.

- 현재 관찰: 멤버십 등급 기준과 요금 캘린더의 우수 고객 할인율 설정이 별도 흐름으로 보입니다.
- 문제: 운영자는 "등급 기준을 바꾸면 객실 할인도 바뀌는가"를 궁금해합니다.
- 개선: 등급 관리 화면에 `할인 연결 상태`를 표시합니다.
- 예시:
  - 등급 기준: 투숙/누적 결제 중 하나 충족
  - 등급별 객실 할인: 요금 캘린더 할인 정책에서 관리
  - 예약 생성 시 적용: 고객 선택 후 견적 계산 단계에서 반영

## 6. 시스템 관점 개선사항

6.1. SYS-01. 날짜 기준을 하나의 영업일 서비스로 고정해야 합니다.

- 현재 관찰: 데이터 감사에서 mock anchor date는 2026-06-10이고 실제 감사일은 2026-06-29로 `dateShiftDays: 19`가 적용됩니다.
- 문제: 페이지별로 오늘 계산, mock data shift, localStorage seed가 섞이면 날짜 회귀가 계속 생깁니다.
- 개선: `PmsDate.today()`, `businessDate`, `rolloverDate`, `mockAnchorDate`를 공통 API로 통일합니다.

6.2. SYS-02. 화면별 localStorage seed를 장기적으로 줄여야 합니다.

- 현재 관찰: storage audit는 통과하지만 55개 페이지가 `pms_currency` 등을 생성하고, 일부 업무 화면은 예약/객실/시재 데이터를 localStorage에 씁니다.
- 문제: 시연 중 새로고침/화면 이동/브라우저 초기화에 따라 화면 간 데이터가 다르게 보일 위험이 있습니다.
- 개선: localStorage는 UI 선호값과 임시 draft 중심으로 제한하고, 업무 데이터는 mock API JSON 또는 공통 store로 이동합니다.

6.3. SYS-03. 예약 상태, 객실 상태, 청소 상태, 정산 상태를 분리 모델로 관리해야 합니다.

- 현재 관찰: 화면에서는 점유 상태와 청소 상태가 함께 표시됩니다.
- 문제: 상태 하나로 모든 것을 표현하려 하면 `예약 없음인데 청소 필요`, `투숙 중인데 방해금지`, `체크아웃 완료인데 미수금` 같은 실제 호텔 케이스가 깨집니다.
- 개선 모델:
  - ReservationStatus
  - OccupancyStatus
  - HousekeepingStatus
  - MaintenanceStatus
  - FolioStatus
  - CashSessionStatus

6.4. SYS-04. 공통 상태/라벨 사전이 필요합니다.

- 현재 관찰: 같은 의미가 `투숙 중`, `재실 중`, `점유 중`, `체크인 완료`처럼 화면별로 다르게 보입니다.
- 개선: `dashboard/common/js/status-catalog.js` 같은 공통 사전을 만들어 라벨, 색상, 아이콘, 우선순위를 함께 관리합니다.

6.5. SYS-05. "다음 액션" 계산 함수를 공통화해야 합니다.

- 문제: 각 페이지가 버튼 노출 조건을 직접 판단하면 미래 체크인 버튼 노출 같은 회귀가 반복됩니다.
- 개선: 예약/객실/folio 조합을 받아 다음 액션을 반환하는 공통 함수가 필요합니다.
- 예시:
  - `getReservationActions(reservation, businessDate)`
  - `getRoomActions(room, activeReservation, housekeeping)`
  - `getFolioActions(folio, reservation)`

6.6. SYS-06. 감사 로그 자동 기록 대상을 함수화해야 합니다.

- 현재 관찰: 감사 로그 화면과 개인정보 감사 구조는 존재합니다.
- 문제: 개별 화면에서 직접 기록하면 누락 가능성이 있습니다.
- 개선: 개인정보 조회, 엑셀 다운로드, 예약 변경, 체크인 후 요금/예치금 변경, 정산 수납, 권한 변경은 wrapper 함수로 기록합니다.

6.7. SYS-07. 단체/회사/일반인 대표 용어 모델을 더 명확히 해야 합니다.

- 현재 관찰: 단체 등록 용어는 개선되었지만 기존 데이터/API에는 company, agency, group이 혼재합니다.
- 개선: 데이터 모델을 다음처럼 분리합니다.
  - `groupAccount`: 단체 등록 주체
  - `representativeType`: 기업, 여행사, 일반인 대표
  - `event`: 실제 행사/투숙 단체
  - `contactPerson`: 단체 담당자
  - `salesOwner`: 호텔 영업 담당자

6.8. SYS-08. 정산과 시재는 데이터 원천을 분리해야 합니다.

- 현재 관찰: 일일 마감은 정산액 검증과 시재 인수인계를 함께 제공합니다.
- 개선: `settlementPayments`는 매출/수납 검증, `cashSessions`는 실제 현금 실사/인수인계로 분리하고, 화면에서만 reconciliation을 계산합니다.

6.9. SYS-09. 등급 기준과 할인 정책을 독립 모델로 분리해야 합니다.

- 현재 관찰: 등급 기준은 멤버십 화면, 할인은 요금 화면에 있습니다.
- 문제: 등급 기준 변경과 할인율 변경이 같은 정책처럼 보일 수 있습니다.
- 개선:
  - `membershipTierCriteria`: 등급 산정 기준
  - `rateDiscountPolicy`: 예약 견적 할인 정책
  - `customerManualOverride`: 수동 등급/할인 예외

6.10. SYS-10. 테스트 결과를 문서와 자동화로 계속 연결해야 합니다.

- 현재 자동화는 좋지만 UX 판단 항목은 수동 리포트에 남아야 합니다.
- 개선: 본 리포트의 `OP/SYS/GUEST/ROAD` 항목 중 구현 완료된 항목은 `USER_ACCEPTANCE_TEST_GUIDE.md`의 회귀 테스트로 추가합니다.

## 7. 투숙객/단체 대표 관점 개선사항

7.1. GUEST-01. 고객 관점에서는 예약 상태명이 더 직관적이어야 합니다.

- 문제: 내부 운영자는 `블록`, `점유 중`, `레이트`를 이해하지만 고객/단체 대표에게는 어렵습니다.
- 개선: 고객에게 보여줄 상태와 내부 상태를 분리합니다.
- 예시:
  - 내부 `block` -> 고객/대표 화면 `객실 배정 대기`
  - 내부 `late` -> 고객/대표 화면 `퇴실 시간 연장`
  - 내부 `in-house` -> 고객/대표 화면 `투숙 중`

7.2. GUEST-02. 단체 대표는 객실 배정과 명단 누락을 한 번에 봐야 합니다.

- 현재 시스템에는 단체 상세 정보가 있으므로 확장 가능성이 좋습니다.
- 개선: 단체 대표용 view 또는 출력 양식에 다음을 제공합니다.
  - 배정 객실 수
  - 대표 등록 여부
  - 동반 투숙객 수
  - 여권/신분증 확인 여부
  - 미정산/예치금/잔액 요약

7.3. GUEST-03. 예치금/잔액/추후 정산 금액은 고객 설명용 문구가 필요합니다.

- 문제: 내부 정산 용어는 고객에게 어렵습니다.
- 개선: 예약 상세/명세서에는 계산 근거를 짧게 표시합니다.
- 예시: `객실요금 ₱1,520 - 예치금 ₱0 = 체크아웃 시 정산 예정 ₱1,520`

7.4. GUEST-04. 부가서비스 주문은 고객 기준 이력으로도 이어져야 합니다.

- 현재는 객실 기준 등록 흐름이 우선입니다.
- 개선: 체크아웃 명세서와 고객 상세에서 POS/골프/렌터카 이용 이력이 같은 주문 ID로 이어지게 합니다.

7.5. GUEST-05. 개인정보 노출은 최소화하고 열람 이력은 고객 문의 대응에 쓸 수 있어야 합니다.

- 현재 감사 로그는 방향이 좋습니다.
- 개선: 고객이 "내 정보를 누가 봤는지" 문의했을 때 관리자 감사 로그에서 대상/직원/시간/사유를 바로 찾을 수 있어야 합니다.

9.1. ROAD-01. 1순위: 상태 사전 통합

- 대상: 예약 현황, 예약 목록, 타임라인, 체크인, 정산, 하우스키핑
- 이유: 사용자 혼란을 가장 많이 줄이고, 이후 기능 추가의 기준이 됩니다.

9.2. ROAD-02. 1순위: 오늘 처리 큐 대시보드

- 대상: 대시보드, 일일 마감
- 이유: PMS의 첫 화면은 예쁜 안내보다 근무자가 놓치면 안 되는 업무를 보여줘야 합니다.

9.3. ROAD-03. 1순위: 다음 액션 CTA 공통화

- 대상: 예약 카드, 예약 목록 행, 타임라인 블록, 단체 상세, 정산 행
- 이유: 같은 상태에서 같은 버튼이 나와야 교육 비용이 줄어듭니다.

9.4. ROAD-04. 2순위: 단체 상세 cockpit화

- 대상: 단체/행사 상세
- 이유: 일반 예약보다 단체 예약은 누락 리스크가 크므로 한 화면 완결성이 중요합니다.

9.5. ROAD-05. 2순위: 정산/시재 데이터 분리 강화

- 대상: folio, night-audit, closing-log
- 이유: 카드 매출과 실제 보유 현금이 섞이면 마감 설명이 불가능해집니다.

9.6. ROAD-06. 2순위: 등급 기준과 할인 정책 연결 안내

- 대상: 멤버십, 요금 캘린더, 예약 생성 견적
- 이유: 운영자가 "등급을 바꾸면 할인도 바뀌는가"를 명확히 알아야 합니다.

9.7. ROAD-07. 3순위: 고객/단체 대표용 출력 뷰

- 대상: 단체 상세, 예약 상세, folio 명세서
- 이유: 내부 PMS 데모를 넘어 실제 고객 응대 품질을 높입니다.

9.8. ROAD-08. 3순위: 채널/외부 예약 포지션 결정

- 대상: 제품 방향
- 이유: Cloudbeds/RMS 같은 PMS는 채널/온라인 예약과도 강하게 연결됩니다. 현재 제품은 내부 운영 PMS에 더 강하므로 포지션을 먼저 정해야 합니다.

## 11. 남은 리스크

11.1. RISK-01. 자동 테스트 통과가 곧 실무 UX 완성은 아닙니다.

버튼이 열리고 콘솔 오류가 없다는 것과, 직원이 빠르게 판단할 수 있다는 것은 다른 문제입니다.

11.2. RISK-02. 상태 기준이 흩어진 상태에서 기능을 계속 추가하면 회귀가 늘어납니다.

청소 상태, 예약 상태, 정산 상태는 먼저 공통화해야 합니다.

11.3. RISK-03. 외부 PMS 벤치마킹은 제품 방향 선택을 요구합니다.

내부 운영형 PMS로 갈지, 채널/온라인 예약까지 포함하는 상용 PMS로 갈지에 따라 우선순위가 달라집니다.

11.4. RISK-04. 정산/마감은 UI보다 데이터 모델이 더 중요합니다.

시재와 매출을 데이터 레벨에서 분리하지 않으면 화면을 아무리 정리해도 설명이 어려워집니다.

## 12. 최종 권고

12.1. 첫 번째 개선 사이클은 새 메뉴 추가가 아니라 상태/액션/오늘 처리 큐 정리로 잡는 것이 좋습니다.

12.2. 두 번째 개선 사이클에서 단체 상세, 정산/마감, 멤버십/요금 할인 연결을 정리합니다.

12.3. 세 번째 개선 사이클에서 고객/단체 대표용 출력 화면과 관리자/감사 기능을 강화합니다.

12.4. 이후 기능 개발 전에는 각 화면의 개선 요구를 `OP`, `SYS`, `GUEST`, `ROAD`, `TASK` 번호로 연결해 작업 티켓화하는 방식이 가장 관리하기 쉽습니다.
