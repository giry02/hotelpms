# Hotel PMS 통합 방향성 및 화면 재구성 리포트

작성일: 2026-06-29  
목적: 기존 벤치마킹/테스트 의견을 중복 없이 통합하고, 어떤 화면은 묶고 어떤 화면은 분리해야 하는지 결정 기준을 제시합니다.

## 1. 최종 방향성

1.1. 결론부터 말하면, 호텔 PMS는 `메뉴별 분리형`만으로 가면 업무 흐름이 끊기고, `한 화면 만능형`으로 가면 권한과 정보 과밀 문제가 생깁니다.

1.2. 가장 적절한 방향은 `호실 중심 운영 cockpit + 업무별 전문 화면 + 권한별 액션 제어`입니다.

1.3. 쉽게 말하면, 프런트가 매일 쓰는 첫 화면은 PC방 좌석 관리처럼 객실/호실을 중심으로 한눈에 보고 처리하게 만듭니다.

1.4. 하지만 결제, 요금 정책, 감사 로그, 직원 권한, 업체/항목 관리처럼 통제와 검증이 필요한 기능은 전문 화면으로 분리합니다.

1.5. 메뉴를 나누는 이유는 "정보를 못 보게 하려고"만이 아니라, 실수하면 안 되는 업무를 별도 흐름으로 잠그기 위해서입니다.

1.6. 따라서 앞으로의 제품 구조는 다음 3층으로 가는 것이 좋습니다.

| 층 | 목적 | 예시 |
| --- | --- | --- |
| 1층: 오늘 운영 화면 | 근무자가 지금 처리할 일을 한 곳에서 봄 | 객실 cockpit, 오늘 체크인/체크아웃, 청소/점검/미수 큐 |
| 2층: 업무 상세 화면 | 특정 업무를 깊게 처리 | 예약 목록, 단체 상세, 정산 목록, 하우스키핑, 시설보수 |
| 3층: 정책/관리 화면 | 기준값, 권한, 마스터 데이터 관리 | 요금 캘린더, 멤버십 기준, 업체/항목 관리, 직원/권한, 감사 로그 |

## 2. PC방식 한 화면 운영에 대한 판단

2.1. PC방식 화면의 장점

- 좌석/호실 단위로 현재 상태를 바로 봅니다.
- 결제, 이용 시간, 좌석 이동, 주문 같은 처리가 한 흐름 안에서 됩니다.
- 초보 직원도 "어느 자리가 어떤 상태인지" 빠르게 이해합니다.
- 화면 이동이 줄어 실무 속도가 빨라집니다.

2.2. 호텔 PMS에 그대로 적용하면 생기는 문제

- 객실에는 투숙객 개인정보가 붙습니다.
- 결제에는 현금/카드/예치금/미수금/환불/단체 정산이 붙습니다.
- 하우스키핑 직원에게 결제 정보를 보여주면 안 됩니다.
- 프런트 직원에게 요금 정책 변경 권한을 주면 위험합니다.
- 시설보수 직원에게 고객 연락처나 folio를 보여줄 필요가 없습니다.
- 단체 예약은 객실 하나가 아니라 행사/명단/정산/대표자 단위로 움직입니다.

2.3. 그래서 정답은 `한 화면에서 다 처리`가 아니라 `한 화면에서 다 보되, 역할별로 다르게 처리`입니다.

2.4. 권장 방식

- 객실 cockpit에서는 상태와 다음 액션을 한눈에 보여줍니다.
- 권한 없는 액션은 숨기거나 읽기 전용으로 보여줍니다.
- 민감 정보는 마스킹하고, 열람 시 감사 로그를 남깁니다.
- 결제/환불/마감 같은 고위험 처리는 cockpit에서 바로 끝내지 않고 전용 drawer 또는 정산 화면으로 넘깁니다.

## 3. 중복 없는 관점 정리

3.1. 시나리오 관점

시나리오 관점은 "사용자가 어떤 일을 끝내려는가"입니다.

| 번호 | 시나리오 | 중심 질문 | 대표 화면 |
| --- | --- | --- | --- |
| SCN-01 | 오늘 객실 운영 | 지금 어떤 객실을 처리해야 하나 | 객실 운영 cockpit |
| SCN-02 | 예약 생성/변경 | 고객이 언제 어떤 객실에 묵나 | 예약 생성/예약 목록 |
| SCN-03 | 체크인 | 오늘 도착 고객을 입실시킬 수 있나 | 객실 cockpit, 체크인 drawer |
| SCN-04 | 재실 관리 | 투숙 중 고객에게 필요한 처리는 무엇인가 | 객실 상세, 부가서비스, 청소, 보수 |
| SCN-05 | 체크아웃/정산 | 퇴실 전 받을 돈과 닫을 folio가 무엇인가 | 정산 drawer, 정산 목록 |
| SCN-06 | 단체 행사 | 객실/명단/대표/정산이 빠짐없이 준비됐나 | 단체 행사 cockpit |
| SCN-07 | 일일 마감 | 오늘 업무를 닫아도 되는가 | 일일 마감 |
| SCN-08 | 기준 관리 | 가격/등급/권한/업체 기준을 어떻게 바꿀 것인가 | 설정/정책 화면 |

3.2. 시스템 및 데이터 관점

시스템 관점은 "데이터가 어디서 오고 어떤 상태로 바뀌는가"입니다.

| 번호 | 데이터 축 | 분리해야 하는 이유 |
| --- | --- | --- |
| DAT-01 | 예약 상태 | 예약 확정, 취소, 노쇼는 고객 계약 상태입니다. |
| DAT-02 | 객실 점유 상태 | 공실, 투숙 중, 체크아웃 예정은 객실 운용 상태입니다. |
| DAT-03 | 청소 상태 | 청소필요, 청소완료, 방해금지는 하우스키핑 상태입니다. |
| DAT-04 | 시설 상태 | 점검 중, 수리 요청, 완료 확인은 객실 판매 가능성과 연결됩니다. |
| DAT-05 | Folio/정산 상태 | 미수금, 예치금, 환불, 정산 완료는 회계 상태입니다. |
| DAT-06 | 현금 시재 상태 | 시작시재, 중간출금, 마감현금은 실제 현금 흐름입니다. |
| DAT-07 | 권한/감사 상태 | 누가 무엇을 보고 바꿨는지 추적합니다. |

3.3. 권한 관점

권한 관점은 "같은 화면을 보더라도 누가 어디까지 할 수 있는가"입니다.

| 번호 | 역할 | 볼 수 있는 것 | 할 수 있는 것 | 제한해야 할 것 |
| --- | --- | --- | --- | --- |
| ROL-01 | 프런트 데스크 | 예약, 객실, 청소 요약, 예치금/잔액 요약 | 체크인, 체크아웃 요청, 객실 이동, 예치금 등록 | 요금 정책 변경, 전체 감사 로그 |
| ROL-02 | 하우스키핑 | 객실 상태, 청소 상태, DND, 요청 메모 | 청소 상태 변경, 보수 요청 | 결제 금액, 고객 연락처 전체 |
| ROL-03 | 시설보수 | 점검/수리 요청, 객실 위치, 문제 유형 | 작업 시작, 완료 요청, 메모 | 결제, 개인정보, 예약 상세 |
| ROL-04 | 회계/정산 | folio, 결제수단, 미수금, 환불, 시재 | 수납 처리, 환불, 마감 검증 | 객실 배정 임의 변경 |
| ROL-05 | 매니저 | 전체 운영 상태, 예외, 감사 | 승인, 조정, 마감 승인 | 없음. 단, 모든 민감 액션은 로그 필요 |
| ROL-06 | 관리자 | 시스템/권한/정책 | 직원/권한/기준 관리 | 실무 화면 직접 처리 최소화 |

3.4. 화면 구조 관점

화면 구조 관점은 "어떤 화면을 합치고, 어떤 화면은 독립시킬 것인가"입니다.

| 번호 | 방향 | 대상 |
| --- | --- | --- |
| UI-01 | 묶어야 함 | 예약 현황 + 객실 상태 + 청소 요약 + 다음 액션 |
| UI-02 | 부분 통합 | 객실 cockpit 안에서 체크인/체크아웃/부가서비스/보수 요청을 drawer로 처리 |
| UI-03 | 별도 유지 | 요금 캘린더, 멤버십 기준, 직원/권한, 감사 로그 |
| UI-04 | 전문 화면 유지 | 정산 목록, 일일 마감, 하우스키핑 작업 목록, 시설보수 작업 목록 |
| UI-05 | cockpit화 | 단체 행사 상세는 별도 화면이지만 내부에서 객실/명단/정산/이력까지 처리 |

## 4. 새 제품 구조 제안

4.1. 핵심 메뉴는 다음처럼 재정리하는 것이 좋습니다.

| 번호 | 새 메뉴 | 포함할 화면 | 성격 |
| --- | --- | --- | --- |
| NAV-01 | 오늘 운영 | 객실 운영 cockpit, 오늘 처리 큐 | 메인 업무 화면 |
| NAV-02 | 예약 관리 | 예약 목록, 예약 타임라인, 예약 생성 | 예약 전문 화면 |
| NAV-03 | 단체/행사 | 단체 목록, 단체 등록, 행사 상세 cockpit | 단체 전문 화면 |
| NAV-04 | 객실 운영 | 하우스키핑, 시설보수, 객실/타입 관리 | 객실 관리 전문 화면 |
| NAV-05 | 부가서비스 | 객실 기준 등록, 업체/항목 관리 | 매출 부가 업무 |
| NAV-06 | 정산/마감 | 정산 목록, 매출 분석, 일일 마감, 마감 로그 | 회계/마감 |
| NAV-07 | 고객/멤버십 | 투숙객, 등급 기준, 등급 이력 | CRM |
| NAV-08 | 설정/감사 | 직원, 권한, 호텔 설정, 감사 로그 | 관리/통제 |

4.2. 이 구조에서 첫 화면은 `오늘 운영`이 되어야 합니다.

4.3. `오늘 운영` 안에는 호실 카드가 있고, 호실 카드에서 바로 열리는 drawer가 있어야 합니다.

4.4. drawer는 권한에 따라 내용이 달라집니다.

## 5. 호실 중심 운영 cockpit 설계

5.1. 객실 카드에서 기본으로 보여줄 정보

| 번호 | 정보 | 이유 |
| --- | --- | --- |
| ROOM-01 | 호실/객실 타입 | 객실 식별 |
| ROOM-02 | 현재 숙박 상태 | 예약 없음, 체크인 예정, 투숙 중, 체크아웃 예정, 점검 중 |
| ROOM-03 | 고객/단체명 | 누가 사용하는 객실인지 확인 |
| ROOM-04 | 청소 상태 | 청소필요, 청소완료, 방해금지 |
| ROOM-05 | 결제 요약 | 잔액 있음/예치금 있음/정산 완료 수준의 요약 |
| ROOM-06 | 부가서비스 합산 | POS/골프/렌터카 금액 요약 |
| ROOM-07 | 다음 액션 | 지금 해야 할 한 가지 주 버튼 |

5.2. 객실 카드에서 바로 처리해도 되는 액션

| 번호 | 액션 | 조건 |
| --- | --- | --- |
| ACT-01 | 체크인 처리 | 프런트 권한, 오늘 입실, 청소완료 또는 경고 승인 |
| ACT-02 | 체크아웃 정산 열기 | 프런트/정산 권한, 체크아웃 대상 |
| ACT-03 | 청소 상태 변경 | 하우스키핑/프런트/매니저 권한 |
| ACT-04 | 부가서비스 추가 | 투숙 중 객실, 프런트/부가서비스 권한 |
| ACT-05 | 보수 요청 | 프런트/하우스키핑/시설보수 권한 |
| ACT-06 | 객실 이동 요청 | 프런트/매니저 권한 |

5.3. 객실 카드에서 바로 처리하면 안 되는 액션

| 번호 | 액션 | 이유 | 처리 방식 |
| --- | --- | --- | --- |
| LOCK-01 | 환불 | 회계/승인 필요 | 정산 화면 이동 또는 승인 drawer |
| LOCK-02 | 요금 정책 변경 | 전체 예약에 영향 | 요금 캘린더/정책 화면 |
| LOCK-03 | 멤버십 기준 변경 | 고객 등급 산정에 영향 | 멤버십 기준 화면 |
| LOCK-04 | 직원 권한 변경 | 보안 영향 | 직원/권한 화면 |
| LOCK-05 | 감사 로그 삭제/수정 | 감사 무결성 훼손 | 불가 또는 최고 관리자만 |
| LOCK-06 | 마감 확정 | 영업일 잠금 영향 | 일일 마감 화면 |

## 6. 묶어도 되는 화면

6.1. GRP-01. 예약 현황 + 객실 상태 + 청소 요약

- 이유: 프런트는 객실을 보면서 예약/상태/청소를 동시에 판단합니다.
- 통합 방식: `객실 운영 cockpit`으로 묶습니다.
- 단, 하우스키핑 상세 작업 목록은 별도로 둡니다.

6.2. GRP-02. 객실 상세 + 부가서비스 등록

- 이유: 부가서비스는 고객이 아니라 실제 투숙 중 객실 기준으로 등록하는 것이 실무에 맞습니다.
- 통합 방식: 객실 drawer 안에 `부가서비스 추가` 탭을 둡니다.
- 업체/항목 관리는 별도 화면으로 유지합니다.

6.3. GRP-03. 객실 상세 + 시설보수 요청

- 이유: 객실 문제는 프런트/하우스키핑이 발견하고 시설보수로 넘기는 흐름이 자연스럽습니다.
- 통합 방식: 객실 drawer 안에서 `보수 요청` 생성.
- 시설보수 팀의 작업 보드는 별도 화면으로 유지합니다.

6.4. GRP-04. 체크인 + 예치금 + 신분증 확인

- 이유: 체크인은 단순 상태 변경이 아니라 입실 가능 조건 확인입니다.
- 통합 방식: 체크인 drawer에서 고객 확인, 예치금, 객실 청소 상태, 특이사항을 함께 처리합니다.

6.5. GRP-05. 체크아웃 + 객실 정산 + 부가서비스 정산

- 이유: 고객은 객실요금과 부가서비스를 따로 느끼지 않고 전체 퇴실 정산으로 이해합니다.
- 통합 방식: 체크아웃 drawer에서 합산 금액을 보여주고, 상세 정산은 정산 목록으로 연결합니다.

6.6. GRP-06. 단체 상세 내부의 객실/명단/정산/이력

- 이유: 단체는 하나의 행사 안에서 객실 배정, 명단, 대표, 정산이 같이 움직입니다.
- 통합 방식: 단체 행사 cockpit을 유지하고 내부 탭으로 처리합니다.

## 7. 따로 둬야 하는 화면

7.1. SEP-01. 요금 캘린더

- 이유: 요금 변경은 미래 예약, 객실 타입, 할인, 매출에 넓게 영향을 줍니다.
- 권장: 운영 cockpit에서 요금 수정은 하지 않습니다. 예약 견적에서는 적용 결과만 보여줍니다.

7.2. SEP-02. 멤버십/등급 기준

- 이유: 등급 기준은 고객 정책이고, 예약 현장에서 자주 바꾸는 값이 아닙니다.
- 권장: 예약/객실 화면에서는 적용된 등급과 할인 근거만 보여줍니다.

7.3. SEP-03. 업체/항목 관리

- 이유: 골프장/렌터카/POS 항목과 바우처 양식은 마스터 데이터입니다.
- 권장: 객실 drawer에서는 이미 등록된 항목만 선택합니다.

7.4. SEP-04. 정산 목록

- 이유: folio, 미수금, 환불, 수납 내역은 회계 검증 화면이 필요합니다.
- 권장: 객실 cockpit에서는 요약과 정산 drawer까지만 제공하고, 복잡한 처리는 정산 목록에서 합니다.

7.5. SEP-05. 일일 마감

- 이유: 영업일을 닫고 롤오버하는 고위험 업무입니다.
- 권장: 마감은 별도 화면으로 유지하되, 오늘 운영/정산에서 미처리 큐가 연결되어야 합니다.

7.6. SEP-06. 감사 로그

- 이유: 감사 로그는 조작 대상이 아니라 확인/추적 대상입니다.
- 권장: 별도 화면으로 유지하고 관리자/매니저 권한으로 제한합니다.

7.7. SEP-07. 직원/권한/호텔 설정

- 이유: 운영 중 실수로 바꾸면 전체 시스템에 영향이 큽니다.
- 권장: 별도 설정 영역으로 유지합니다.

## 8. 권한 문제 해결 방식

8.1. 권한은 메뉴 분리만으로 해결하면 안 됩니다.

8.2. 같은 호실 cockpit을 보더라도 역할별로 다음이 달라져야 합니다.

| 번호 | 제어 방식 | 설명 |
| --- | --- | --- |
| PERM-01 | 메뉴 접근 권한 | 특정 전문 화면 자체를 숨김 |
| PERM-02 | 화면 내 섹션 권한 | 결제/개인정보/감사 섹션을 숨김 |
| PERM-03 | 액션 권한 | 버튼을 숨기거나 비활성화 |
| PERM-04 | 필드 권한 | 연락처, 이메일, 결제 상세를 마스킹 |
| PERM-05 | 승인 권한 | 환불, 마감, 요금 변경은 승인 필요 |
| PERM-06 | 감사 기록 | 민감 열람/변경은 자동 기록 |

8.3. 예시: 하우스키핑 직원이 객실 cockpit을 볼 때

- 볼 수 있음: 호실, 청소 상태, DND, 체크아웃 예정 여부, 요청 메모
- 할 수 있음: 청소필요/청소완료/DND 변경, 보수 요청
- 볼 수 없음: 고객 연락처, 결제 금액, 예치금, folio 상세

8.4. 예시: 프런트 직원이 객실 cockpit을 볼 때

- 볼 수 있음: 예약자, 단체명, 입퇴실, 청소 상태, 예치금/잔액 요약
- 할 수 있음: 체크인, 체크아웃 요청, 부가서비스 등록, 객실 이동 요청
- 제한: 환불, 요금 정책 변경, 마감 확정

8.5. 예시: 회계 담당자가 객실 cockpit을 볼 때

- 볼 수 있음: 정산 대상 객실, folio, 결제수단, 미수금
- 할 수 있음: 수납, 환불 요청/처리, 정산 완료
- 제한: 청소 상태 변경, 객실 배정 임의 변경

## 9. 추천 화면 구성안

9.1. 첫 화면: 오늘 운영

구성:

- 상단: 오늘 체크인, 체크아웃, 미도착, 미수금, 청소필요, 점검중, 단체 도착
- 중앙: 객실 그리드
- 오른쪽 또는 drawer: 선택 객실 상세
- 하단 또는 접힘: 긴급 알림, 인수인계 메모

9.2. 객실 drawer 탭

| 번호 | 탭 | 주요 내용 |
| --- | --- | --- |
| DRW-01 | 요약 | 고객/단체, 입퇴실, 상태, 메모 |
| DRW-02 | 체크인/체크아웃 | 신분증, 예치금, 잔액, 정산 CTA |
| DRW-03 | 청소 | 청소 상태, DND, 하우스키핑 메모 |
| DRW-04 | 부가서비스 | POS/골프/렌터카 등록 및 합산 |
| DRW-05 | 보수 | 요청 생성, 진행 상태 |

- drawer 안에서 1분 안에 끝나는 단순 처리는 즉시 처리합니다.
- 승인/검증/복수 행 편집/정책 변경이 필요한 업무는 전문 화면으로 이동합니다.

## 11. 데이터 모델 방향

11.1. cockpit이 한 화면처럼 보여도 데이터는 하나로 합치면 안 됩니다.

11.2. 화면은 통합하고, 데이터 모델은 분리하는 것이 핵심입니다.

## 12. 구현 로드맵

12.1. STEP-01. 공통 상태 사전부터 만듭니다.

- 예약 상태, 객실 상태, 청소 상태, 정산 상태의 라벨/색상/아이콘을 통합합니다.
- 이 작업 없이는 cockpit을 만들어도 화면마다 표현이 흔들립니다.

12.2. STEP-02. `오늘 운영` cockpit의 정보 설계를 먼저 합니다.

- 기존 예약 현황을 기반으로 하되, 결제/청소/보수/부가서비스는 요약만 붙입니다.
- 모든 기능을 한 번에 넣지 말고 drawer 탭 구조로 확장합니다.

12.3. STEP-03. 권한별 표시/액션 매트릭스를 적용합니다.

- 먼저 프런트, 하우스키핑, 회계, 매니저 4개 역할만 적용해도 충분합니다.
- 이후 시설보수, 관리자, 단체 영업 담당을 세분화합니다.

12.4. STEP-04. 정산과 마감은 전문 화면으로 유지하되 연결성을 강화합니다.

- 객실 cockpit에서 `정산 열기`를 누르면 해당 folio가 열린 상태로 정산 화면 또는 drawer가 열려야 합니다.
- 일일 마감의 미처리 큐는 해당 업무 화면으로 필터를 물고 이동해야 합니다.

12.5. STEP-05. 단체 행사 cockpit은 별도 고도화합니다.

- 단체는 호실 중심 cockpit만으로 부족합니다.
- 행사 단위로 객실, 명단, 대표, 정산, 변경 이력을 한 화면에서 봐야 합니다.

## 13. 최종 권고

13.1. 메뉴를 모두 합치는 방향은 추천하지 않습니다.

13.2. 대신 `오늘 운영`이라는 강한 메인 화면을 만들고, 그 안에서 객실 단위로 대부분의 일상 업무를 시작하게 해야 합니다.

13.3. 하우스키핑/시설보수/정산/요금/감사/설정은 전문 화면으로 유지하되, cockpit에서 필요한 순간 바로 연결되게 합니다.

13.4. 권한 문제는 화면을 쪼개서 해결하기보다, 같은 화면 안에서 섹션/버튼/필드/승인/감사 로그를 제어하는 방식으로 해결하는 것이 좋습니다.

13.5. 제품 방향을 한 문장으로 정리하면 다음과 같습니다.

> Hotel PMS는 호실을 중심으로 오늘의 운영을 한 화면에서 처리하되, 결제·정책·감사·마감 같은 고위험 업무는 권한과 전문 화면으로 통제하는 운영 cockpit형 PMS로 가는 것이 가장 적합합니다.
