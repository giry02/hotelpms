# Hotel PMS 전 과정 단일 사이클 E2E 시나리오

## 1. 목적

초기 데이터가 없는 전용 QA 호텔에서 기준정보를 생성하고, 개인 예약과 단체 예약을 실제 업무 순서대로 처리한 뒤 부가서비스, 비품 구매 지출, 정산, 체크아웃, 하우스키핑, 일일 마감까지 하나의 데이터 흐름으로 검증한다.

이 시나리오는 화면이 열리거나 버튼이 눌리는지만 확인하지 않는다. 각 단계에서 생성된 ID, 상태, 금액, 통화, 객실, 투숙객이 다음 화면과 새로고침 후에도 동일한지 검증한다.

## 2. 실행 안전 기준

- 운영 공용 호텔 `The Grand Saigon`의 데이터를 삭제하지 않는다.
- 전용 QA 테넌트 `TENANT-E2E-CYCLE-001` 또는 격리된 Playwright 브라우저 컨텍스트에서만 초기화한다.
- 초기화 전 저장소/API 스냅샷을 내보내고 실행 ID를 기록한다.
- 삭제 범위는 테스트 테넌트의 예약, 고객, 단체, 객실, 부가서비스, Folio, 결제, 지출, 시재, 감사 로그로 제한한다.
- 기준 샘플 JSON이 자동 재주입되는 환경에서는 단순 `localStorage.clear()`를 초기화 성공으로 판정하지 않는다. 초기화 직후 각 목록 API가 0건인지 확인한다.
- 다른 테넌트의 데이터 수가 변하면 즉시 중단하고 P0 결함으로 기록한다.

## 6. 단계별 상세 절차

### CYCLE-000 전용 QA 데이터 초기화

1. QA 테넌트로 로그인한다.
2. 실행 전 예약·객실·고객·단체·주문·Folio·지출·감사 로그 건수를 기록한다.
3. QA 테넌트 업무 데이터를 초기화한다.
4. 모든 목록 API와 화면을 새로고침한다.

PASS 기준:

- 업무 데이터 목록이 모두 0건이다.
- 호텔 설정과 관리자 계정처럼 테스트 실행에 필요한 최소 인증 정보만 남는다.
- 다른 테넌트의 전후 건수와 체크섬이 같다.
- 초기화 후 기존 데모 데이터가 다시 나타나지 않는다.

### CYCLE-005 Admin 입점과 PMS 연결 계약

1. Admin 입점 신청 화면에서 `QA Cycle Hotel`을 신청한다.
2. Admin 목록·상세에서 신청값과 승인 상태를 확인한다.
3. 승인된 테넌트 식별자를 PMS 호텔 식별자와 대조한다.

PASS 기준:

- 신청값은 Admin 목록과 새로고침 후 상세에서 동일하다.
- 승인·반려 이력과 담당자가 감사 로그에 남는다.
- 서버 API가 없는 정적 배포 환경에서는 Admin 저장소와 PMS 저장소가 분리되어 있으므로 실제 자동 전파를 PASS로 처리하지 않는다. 이 경우 `BLOCKED(아키텍처)`로 기록한다.

### CYCLE-010 기준정보 생성

1. 호텔 기준 통화를 PHP로 설정하고 시작 시재 PHP 10,000을 등록한다.
2. 객실 유형 `Cycle Deluxe`, 1박 PHP 2,000을 등록한다.
3. 객실 `QA-101`, `QA-102`, `QA-201`, `QA-202`를 생성한다.
4. 네 객실을 `공실 + 청소 완료`로 만든다.
5. 통합 POS `Welcome Snack PHP 320`, 음식점 `Dinner Set PHP 680`, 골프 `18H Green Fee PHP 900`, 렌터카 `Airport Van PHP 960`, 기타 `Late Arrival Kit PHP 200`을 등록한다.

PASS 기준:

- 객실 목록, 예약 현황, 업체/항목 관리에 같은 기준정보가 표시된다.
- 객실번호와 업체/항목 ID가 중복 생성되지 않는다.
- 새로고침 후 금액과 통화가 유지된다.

### CYCLE-012 역할·기능 권한과 계정 상태

1. 프런트 역할에서 `정산 취소`, `부가서비스 취소` 기능 권한을 해제한다.
2. 프런트 계정으로 두 되돌리기 동작이 차단되는지 확인한다.
3. 관리자 계정으로 권한을 부여하고 다시 로그인해 동작을 확인한다.
4. 테스트 직원을 휴직·퇴직으로 각각 변경해 로그인을 시도한다.

PASS 기준:

- 기능 권한은 직원 목록이 아닌 권한 설정에서 역할별로 저장된다.
- 권한 없는 사용자는 버튼 미노출 또는 명시적 차단이며 데이터·감사 로그가 변하지 않는다.
- 권한 부여 후에는 동작과 감사 로그가 각각 1건이다.
- 휴직·퇴직 계정은 로그인할 수 없고 근무 중 계정만 로그인할 수 있다.

### CYCLE-015 객실 점검 차단과 복구

1. `QA-102`를 점검 중으로 전환한다.
2. 신규 예약과 체크인을 시도해 차단 메시지와 상태 불변을 확인한다.
3. 점검을 완료하고 객실이 청소 필요로 전환되는지 확인한다.
4. 청소 완료 후 예약 가능 목록에 다시 나타나는지 확인한다.

PASS 기준:

- 점검 중에는 예약·체크인이 저장되지 않는다.
- 점검 종료 직후 상태는 청소 완료가 아니라 청소 필요다.
- 청소 완료 후에만 판매 가능하며 다른 객실 상태는 변하지 않는다.

## 8. 단계 중단 기준

다음 중 하나라도 발생하면 후속 단계로 넘어가지 않는다.

- 성공 메시지가 표시됐지만 새로고침 후 데이터가 사라짐
- 선택한 객실과 저장된 객실이 다름
- 다른 예약·객실·테넌트 데이터가 변경됨
- 동일 조작으로 저장·감사 로그가 2건 이상 생성됨
- 개인·단체·부가서비스·정산 금액 중 하나라도 기대값과 다름
- 지출이 매출에 양수로 합산되거나 카드 결제가 현금 시재에 반영됨
- 체크아웃 완료 예약이 미도착·레이트·투숙 중으로 되돌아감

## 10. 자동화 구현 단위

1. `resetQaTenant()` - 허용된 테스트 테넌트만 초기화하고 전후 체크섬 비교
2. `seedHotelMaster()` - 호텔·직원·객실·업체/항목 최소 기준정보 생성
3. `runIndividualStayCycle()` - 개인 예약부터 체크아웃까지
4. `runExpenseCycle()` - 비품 구매 CRUD, 정산·시재 반영
5. `runGroupStayCycle()` - 단체 업체·행사·배정·명단·체크인·부가서비스
6. `runSettlementAndCloseCycle()` - 수납, 정산, 체크아웃, 하우스키핑, 마감
7. `verifyCrossModuleState()` - 각 ID와 금액을 모든 연계 화면/API에서 대조
8. `exportCycleEvidence()` - 단계별 스크린샷, 콘솔, API, 저장소 스냅샷, 결과 JSON 생성

자동화는 하나의 브라우저 컨텍스트와 동일 실행 ID를 유지해야 한다. 각 기능을 별도 컨텍스트에서 독립 fixture로 실행하면 이 시나리오의 PASS로 인정하지 않는다.

## 11. 실행 결과 판정 계층

| 계층 | 의미 | 최종 사이클 PASS 반영 |
|---|---|---|
| `CONNECTED_FLOW` | 같은 브라우저 컨텍스트 안에서 앞 단계가 만든 ID를 다음 단계가 사용 | 가능 |
| `ISOLATED_FUNCTIONAL` | 기능별 격리 fixture로 CRUD·권한·예외를 검증 | 보조 증거만 가능 |
| `CONTRACT` | Admin/PMS처럼 저장소가 분리된 경계의 필드·상태 계약 검증 | 보조 증거만 가능 |
| `BLOCKED(아키텍처)` | 서버 API/공유 DB가 없어 실제 전파를 검증할 수 없음 | 불가 |

배포 사이트 전체 회귀가 모두 PASS여도 `CONNECTED_FLOW`의 CYCLE-000~140이 동일 실행 ID로 끝까지 성공하지 않으면 “전체 단일 사이클 PASS”라고 보고하지 않는다.
