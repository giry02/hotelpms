- 문서 버전: 1.0
- 작성일: 2026-07-21
- 테스트 대상: `https://hotelpms-eight.vercel.app`
- 지원 화면: 데스크톱, 태블릿
- 관련 문서:
  - `PMS_PHASE1_PROCESS_TEST_EXECUTION_PLAN_20260720.md`
  - `PMS_COMPLETE_INTEGRATION_TEST_MASTER_PLAN_20260720.md`

## 1. 목적

기존 계획은 확정된 업무 프로세스가 정상 동작하는지 확인하는 회귀 테스트다. 이 문서는 이미 알려진 정상 경로를 다시 확인하는 데 그치지 않고, 아직 발견되지 않은 결함을 찾기 위한 탐색 테스트 기준을 정의한다.

다음 유형을 우선 탐색한다.

1. 같은 데이터가 진입 경로, 필터, 화면에 따라 다르게 보이는 결함
2. 버튼은 동작하지만 저장, 새로고침, 재로그인 후 상태가 되돌아가는 결함
3. 날짜, 통화, 객실 상태, 권한 조합에 따라 잘못된 분기로 진입하는 결함
4. 빠른 반복 클릭, 뒤로 가기, 새로고침, 다중 탭에서 발생하는 결함
5. 데이터가 누적됐을 때 전체 데이터를 매번 읽거나 카운트와 목록이 달라지는 결함
6. 언어 변경 후 일부 텍스트만 이전 언어로 남거나 화면이 잠시 잘못 표시되는 결함
7. 태블릿에서 버튼, 셀렉트, 팝업이 잘리거나 클릭 대상이 어긋나는 결함

## 2. 기존 테스트와의 차이

| 구분 | 회귀 테스트 | 잔여 결함 탐색 테스트 |
|---|---|---|
| 목적 | 명시된 요구사항의 정상 동작 확인 | 명시되지 않은 조합·경계·불일치 발견 |
| 입력 | 고정 fixture 중심 | 상태·날짜·권한·언어·진입 경로 조합 |
| 판정 | 예상 결과 일치 여부 | 중간 상태, 영속성, 교차 화면, 로그까지 일치 여부 |
| 반복 | 동일 경로 반복 | 같은 작업을 서로 다른 경로와 순서로 반복 |
| 결과 | PASS/FAIL | PASS/FAIL/BLOCKED와 신규 결함 ID |

## 3. 절대 판정 규칙

### 3.1 PASS 필수 조건

한 케이스는 아래 항목을 모두 만족해야 PASS다.

1. 지정한 실제 배포 URL에서 실행했다.
2. 지정한 입력값과 순서를 그대로 사용했다.
3. 버튼 클릭 직후의 중간 상태가 예상과 일치했다.
4. 저장 후 목록과 상세 팝업이 같은 값을 표시했다.
5. 새로고침 후에도 값과 상태가 유지됐다.
6. 연결된 다른 화면의 동일 데이터가 일치했다.
7. 상태 변경 작업은 운영 감사 로그에 정확히 1건 기록됐다.
8. 취소·차단 경로는 데이터와 감사 로그를 생성하지 않았다.
9. 테스트 데이터와 변경한 기준 데이터를 원복했다.
10. 필수 증거 파일이 남아 있다.

### 3.2 PASS로 인정하지 않는 경우

- 페이지가 열렸다는 이유만으로 PASS 처리
- 버튼이 눌렸다는 이유만으로 PASS 처리
- 토스트 메시지만 보고 저장 성공으로 판단
- 자동 테스트 통과만으로 실제 화면 테스트를 생략
- 새로고침, 재진입, 교차 화면 확인 없이 PASS 처리
- 지정 fixture와 다른 데이터를 사용하고 기록하지 않음
- 테스트 중 생성한 데이터를 남긴 채 PASS 처리

### 3.3 결과 상태

| 상태 | 기준 |
|---|---|
| PASS | 모든 필수 조건과 증거를 충족 |
| FAIL | 실제 결과가 예상 결과와 다름 |
| BLOCKED | 외부 배포, 계정, 데이터 훼손 등으로 실행 자체가 불가능 |
| NOT RUN | 아직 실행하지 않음. 최종 완료 보고서에는 허용하지 않음 |

## 4. 실행 환경과 증거

### 4.1 환경

| 항목 | 기준 |
|---|---|
| 배포 URL | `https://hotelpms-eight.vercel.app` |
| 브라우저 | Chromium 최신 버전 |
| 데스크톱 | 1440×900 이상 |
| 태블릿 | 820×1180, 1024×1366 |
| 언어 | 한국어, 영어 |
| 계정 | 관리자, 프런트 데스크, 권한 제한 역할 |
| 시간 기준 | 실행 당일의 Asia/Seoul 날짜. 화면 고정 날짜 사용 금지 |

모바일은 지원 범위가 아니므로 기능 PASS 대상으로 판정하지 않는다. 다만 모바일 너비에서 강제 리디렉션 또는 지원 안내가 정의돼 있다면 해당 동작만 확인한다.

### 4.2 테스트 데이터 규칙

- 생성 데이터 이름은 `DISC-YYYYMMDD-{CASE-ID}`로 시작한다.
- 기존 고객, 예약, 단체, 직원 데이터는 삭제하지 않는다.
- 기존 데이터를 변경할 때는 실행 전 값을 증거에 기록하고 종료 시 원복한다.
- 객실 상태, 청소 상태, 정산 상태를 변경하면 원래 상태와 원복 결과를 모두 캡처한다.
- 각 케이스는 단독 실행 가능해야 하며 이전 케이스의 성공에 의존하지 않는다.

## 5. 실행 방식

### 5.1 한 케이스의 표준 순서

1. 화면 진입 전 언어, 역할, 현재 날짜, 대상 데이터 상태를 기록한다.
2. 대상 화면을 직접 URL로 열고 초기 상태를 캡처한다.
3. 사용자와 같은 방식으로 입력, 선택, 저장, 취소를 실행한다.
4. 클릭 직후 모달 닫힘, 토스트, 카드 상태, 카운트를 확인한다.
5. 상세 팝업을 다시 열어 저장값을 확인한다.
6. 브라우저 새로고침 후 같은 값을 확인한다.
7. 예약 목록, 타임라인, 객실 관리, 정산, 감사 로그 등 연결 화면을 확인한다.
8. 같은 작업을 메뉴 진입 경로와 직접 URL 진입 경로에서 비교한다.
9. 테스트 데이터를 삭제하거나 원래 상태로 되돌린다.
10. 원복 결과를 새로고침 후 확인하고 결과를 기록한다.

### 5.2 실행 우선순위

| 단계 | 내용 | 예상 시간 | 중단 조건 |
|---|---|---:|---|
| 1 | 예약·객실 상태 머신 | 3시간 | P0/P1 결함 발견 시 수정·재배포 후 재시작 |
| 2 | 교차 화면·카운트·필터 일치 | 2시간 | 동일 데이터 불일치 발견 시 관련 화면 묶음 재시험 |
| 3 | 단체·부가서비스·정산 연결 | 3시간 | 금액 또는 상태 영속성 결함 발견 시 정산 케이스 중단 |
| 4 | 경계값·오류 입력·비정상 조작 | 2시간 | 데이터 훼손 가능성이 있으면 즉시 중단 |
| 5 | 다국어·태블릿·시각 안정성 | 2시간 | 클릭 불가 또는 화면 가림 발생 시 기능 테스트 우선 수정 |
| 6 | 데이터량·성능 | 2시간 | 7초 초과 또는 전체 이력 무제한 로드 발견 시 FAIL |

## 7. 결함 등급과 처리

| 등급 | 기준 | 처리 |
|---|---|---|

## 8. 수정 후 재시험 규칙

1. 실패한 원본 케이스를 같은 데이터와 순서로 재실행한다.
2. 같은 화면의 인접 상태와 반대 분기를 실행한다.
3. 연결된 페이지에서 동일 데이터가 일치하는지 확인한다.
4. 한국어·영어, 데스크톱·태블릿을 각각 확인한다.
5. 자동 회귀 스위트를 실행하되 자동 통과를 수동 실행의 대체로 사용하지 않는다.
6. 배포 커밋과 Vercel 반영을 확인한 뒤 최종 PASS를 부여한다.
7. 결함 재현 증거, 수정 커밋, 수정 후 증거를 같은 폴더에 보존한다.

## 9. 완료 기준

다음 조건을 모두 충족해야 잔여 결함 탐색 테스트 완료로 보고한다.

- 본문 63개 케이스가 모두 PASS 또는 승인된 예외 상태다.
- NOT RUN과 BLOCKED가 0건이다.
- P0, P1 미해결 결함이 0건이다.
- P2 결함은 모두 수정·재시험했거나 사용자 승인으로 이관됐다.
- 테스트 생성 데이터와 변경 fixture가 원복됐다.
- 결과 파일, 스크린샷, 결함 목록, 수정 커밋이 연결됐다.
- 새로 발견된 결함은 영구 회귀 케이스로 문서화됐다.

단순 클릭 성공과 업무 상태 전이 성공을 구분한다. 아래 케이스는 예약 처리 버튼의 존재 여부가 아니라 예약, 객실, 하우스키핑, 감사 로그, 모달, 저장소가 함께 바뀌는지 확인한다.

| ID | 사전 상태 | 실행 | 합격 기준 |
|---|---|---|---|
| UREG-RES-001 | 투숙 중, 미수금 PHP 25 | 체크아웃 | 예약·객실 불변, 모달 유지, 금액 안내, 감사 로그 0건 |
| UREG-RES-002 | 예정 종료일이 미래인 투숙객 | 조기 체크아웃 확인에서 취소 | 투숙 상태 유지, 객실 점유 유지, 감사 로그 0건 |
| UREG-RES-003 | 금일 도착, 객실 청소 필요 | 체크인 경고에서 취소 | 예약 확정·객실 청소 필요 유지, 모달 유지, 저장·감사 로그 0건 |
| UREG-RES-004 | 금일 도착, 객실 청소 필요 | 경고 확인 후 체크인 | 예약 투숙 중, 객실 점유, 청소 필요 경고 상태 유지, 저장·감사 로그 각 1건, 모달 닫힘 |
| UREG-RES-005 | 객실 이동 이력이 있는 투숙객 | 체크아웃 | 현재 객실만 공실·청소 필요로 변경, 이전 객실 불변 |
| UREG-RES-006 | 익일 도착 확정 예약 | 체크인 | 예약·객실 불변, 모달 유지, 감사 로그 0건 |
| UREG-RES-007 | 투숙 중, 미수금 없음 | 체크아웃 버튼 동시 2회 실행 | 확인·저장·하우스키핑 동기화·감사 로그가 각각 정확히 1회 |
| UREG-RES-008 | 체크아웃 완료 예약 | 체크아웃 재실행 | 상태 불변, 저장·감사 로그 추가 0건 |
| UREG-RES-009 | 투숙 중, 브라우저 저장소 폴백 | 체크아웃 후 새로고침 | 완료 시각과 공실·청소 필요 상태가 저장소와 새로고침 후 모두 동일 |

각 케이스는 수정 전 실패 증거와 수정 후 PASS를 함께 보존한다. 한 케이스에서 결함을 발견하면 동일 화면의 반대 분기와 인접 상태를 최소 2개 이상 추가 실행한다.

## 13. 교차 도메인·반복 조작 영구 회귀 케이스

한 화면의 저장 성공만으로 PASS 처리하지 않는다. 예약, 객실, 부가서비스, 정산, 지출, 권한, 감사 로그처럼 동일 업무 ID를 공유하는 데이터는 연결된 저장소와 재진입 결과까지 확인한다.

| ID | 실제 입력·조작 | 합격 기준 |
|---|---|---|
| STATE-001 | 투숙 예약을 1201→1202→1203으로 연속 객실 이동 | 최종 예약 호실 1203, 이전 객실 공실, 현재 객실 점유, 이동 이력·감사 로그 2건 |
| STATE-002 | 동일 정산을 완료→미완료로 2회 반복 | 최종 미완료, 완료 레코드 잔존 없음, 미완료 전환 감사 로그 정확히 2건 |
| STATE-003 | 동일 부가서비스를 완료→미완료로 2회 반복 | 최종 미완료, 주문 저장값 일치, 미완료 전환 감사 로그 정확히 2건 |
| STATE-004 | 현금 지출 등록→카드 변경→현금·금액 변경 | 지출 원장과 시재 출금이 중복 없이 최종 결제수단·금액으로 일치 |
| STATE-005 | 예약 편집 중 메모 입력 후 한국어→영어 전환 | 미저장 입력 유지, 화면 라벨 즉시 영어 전환, 예약 원본 미변경 |
| STATE-006 | 권한 없는 부가서비스 미완료 전환 후 권한 부여 재시도 | 최초 차단·저장 불변, 권한 부여 후 저장 및 감사 로그 1건 |
| STATE-007 | 권한 없는 정산 미완료 전환 후 권한 부여 재시도 | 최초 차단·완료 유지, 권한 부여 후 미완료 저장 및 감사 로그 1건 |
| STATE-008 | 금일 체크인 예약의 플랫카드 항공편 입력·저장·인쇄 | 입력 즉시 미리보기 반영, 재진입 유지, A4 가로 인쇄 규칙 적용 |
| STATE-009 | 예약 현황의 모든 필터 순회 | 버튼 카운트와 카드 수 일치, 동일 예약은 필터별 상태·색상 불변 |
| STATE-010 | 부가서비스가 연결된 투숙 예약을 다른 객실로 이동 | 예약 ID 불변, 부가서비스 주문의 room/roomNo/roomId/fullRoom이 새 객실로 함께 이동 |
| STATE-011 | 정산 원장이 연결된 투숙 예약을 다른 객실로 이동 | 예약 ID가 같은 Folio의 room/roomNo/roomId가 새 객실로 함께 이동하고 금액·상태는 불변 |
| STATE-012 | 현금 지출 등록 후 해당 지출 삭제 | 지출 원장에서 삭제되고 연결된 시재 출금이 정확히 1건 역분개되며 잔액이 복원 |
| STATE-013 | PHP 지출을 USD로 통화·금액 수정 | 이전 PHP 집계에서 제거되고 USD 집계에 최종 금액만 1회 반영 |
| STATE-014 | 지출·시재 저장소가 손상된 상태로 페이지 진입 | 런타임 오류 없이 빈 목록으로 복구 표시하고 손상 원본을 묵시적으로 덮어쓰지 않음 |
| STATE-015 | 예정 종료일보다 일찍 투숙객 체크아웃 | 예약은 완료 상태로 유지되고 실제 퇴실 시각이 기록되며 객실은 공실·청소 필요로 전환되고 모달이 닫힘 |
| STATE-016 | 체크아웃 청소 작업을 완료한 뒤 새로고침 | 객실의 status/frontStatus/housekeepingStatus/cleaningStatus가 모두 공실·청소 완료로 저장되고 작업 완료 상태 유지 |
| STATE-017 | 예약 현황에서 1203호를 선택한 뒤 체크인·체크아웃 날짜 변경 | 선택 객실이 1203호로 유지되고 다른 빈 객실로 자동 변경되지 않음 |
| STATE-018 | 동일 투숙객의 체크아웃 처리를 동시에 2회 요청 | 예약 완료와 객실 해제는 한 번만 반영되고 체크아웃 감사 로그도 정확히 1건 |
| STATE-019 | 청소 필요 객실 체크인 경고를 취소한 뒤 다시 확인 | 취소 시 예약·객실·로그 불변, 확인 시 투숙 중·점유/청소 필요로 저장되고 감사 로그 1건 |

STATE-010과 STATE-011은 화면별 단독 테스트로는 놓치기 쉬운 교차 도메인 결함을 방지한다. STATE-015~019는 날짜 경계, 중복 요청, 객실 선택 보존, 체크아웃 이후 하우스키핑 복구를 검증한다. 이후 예약이나 청소 상태 로직을 변경할 때 이 19개 상태 전이를 모두 재실행한다.
