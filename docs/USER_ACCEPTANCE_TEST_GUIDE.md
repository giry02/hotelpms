# Hotel PMS 사용자 테스트 / 통합 테스트 가이드

작성일: 2026-06-25  
목적: 반복 발생한 테스트 누락을 기록하고, 이후 모든 화면 수정 후 동일 기준으로 사용자 테스트, 통합 테스트, 데이터 적합성 테스트를 수행하기 위한 기준 문서입니다.

## 1. 반복 실수 기록

아래 항목은 실제 작업 중 반복적으로 놓친 내용입니다. 이후 검수 시 우선순위를 가장 높게 둡니다.

| 구분 | 놓친 내용 | 영향 | 앞으로의 검증 기준 |
| --- | --- | --- | --- |
| 날짜 기준 | 오늘, 금일, 주간, 월간 기준이 화면마다 다르게 적용됨 | 시연일 기준 데이터가 맞지 않아 신뢰 저하 | 모든 화면은 `PmsDate.today()` 또는 동일 영업일 기준을 사용하고, 금일 필터 결과가 실제 기준일과 일치해야 함 |

| 정산 데이터 | 정산 목록, 매출 분석, 일일 마감 금액이 서로 다름 | 운영 기준을 설명하기 어려움 | 동일 Revenue/Folio/Settlement 소스를 기준으로 합계, 미수금, 예치금, 결제수단을 대조 |

| 팝업/상세 | 목록의 정보와 팝업 상세 정보가 다르거나 부족함 | 운영자가 실제 처리 판단을 못함 | 목록 클릭, 상세 버튼, 팝업의 데이터가 같은 엔티티를 가리키는지 확인 |

| 샘플 데이터 | 시연에 필요한 케이스가 부족함 | 버튼/필터/상태를 확인할 수 없음 | 완료/미완료/미수금/블랙리스트/레이트 체크아웃/변경 이력 등 필수 케이스 포함 |

## 2. 테스트 원칙

1. 수정한 화면만 보지 않고 연관 화면까지 확인합니다.
2. 데이터는 화면에 “보이는지”가 아니라 업무적으로 “말이 되는지”를 검증합니다.
3. 오늘, 금일, 체크인, 체크아웃, 마감일은 항상 실제 영업일 기준으로 검증합니다.
4. 호텔 운영자가 20년 경력자라고 가정하고, 버튼명과 상태명이 오해 없이 읽히는지 확인합니다.
5. 고객이 입력한 데이터와 시스템 라벨을 구분합니다. 객실 유형명, 고객명, 업체명은 임의 번역하지 않습니다.
6. GitHub Pages에서 확인하는 것을 최종 기준으로 둡니다. 로컬 통과만으로 완료하지 않습니다.

## 3. 수정 후 필수 실행 순서

### 3.1 정적 검증

```powershell
git diff --check
npm run audit:data
npm run audit:storage
```

확인 기준:

- JSON 파싱 오류가 없어야 합니다.
- 화면 데이터가 서로 다른 mock source에서 불일치하지 않아야 합니다.
- localStorage에만 의존해 새로고침 후 사라지는 핵심 데이터가 없어야 합니다.

### 3.2 페이지 스모크 테스트

```powershell
npm run smoke
npm run smoke:all
```

확인 기준:

- 주요 페이지가 열립니다.
- 콘솔 오류가 없어야 합니다.
- 한국어/영어 언어 전환 후 키 문자열이 그대로 노출되지 않아야 합니다.
- 검색, 필터, 정렬, 페이지네이션 기본 동작이 끊기지 않아야 합니다.

### 3.3 상호작용 테스트

```powershell
npm run smoke:interactions
npm run e2e
npm run test:reservations
npm run test:groups
```

확인 기준:

- 목록 클릭 시 상세/관리 팝업이 열립니다.
- 버튼 클릭 후 상태 변경이 화면과 데이터에 함께 반영됩니다.
- 예약, 단체, 정산, 하우스키핑, 시설보수처럼 연결된 업무 흐름은 다음 화면에 결과가 이어져야 합니다.

### 3.4 시각 레이아웃 테스트

```powershell
npm run audit:visual
```

확인 기준:

- 검색 영역, 필터, 정렬 버튼 간격이 기준 화면과 일관됩니다.
- 테이블이 가로 스크롤을 만들지 않습니다. 단, 명시적으로 필요한 대형 표는 예외입니다.
- 팝업 제목, 버튼, 하단 액션 영역이 깨지지 않습니다.
- 텍스트가 버튼이나 카드 밖으로 넘치지 않습니다.

### 3.5 GitHub Pages 최종 검증

수정 후 반드시 커밋과 푸시를 먼저 수행합니다.

```powershell
git status --short
git add <changed-files>
git commit -m "<message>"
git push origin main
git ls-remote origin main
```

그 다음 GitHub Pages URL에 cache busting 값을 붙여 확인합니다.

```text
https://giry02.github.io/hotelpms/<page-path>?cb=<commit-hash>
```

확인 기준:

- raw GitHub와 GitHub Pages가 모두 새 커밋 내용을 포함해야 합니다.
- 브라우저 localStorage 캐시 때문에 예전 데이터가 보이는 경우 API/mock data 버전을 갱신하거나 초기화 루틴을 확인합니다.

## 4. 업무 영역별 필수 검수 항목

### 4.1 예약 타임라인 / 예약 관리

- 객실 정렬 기준이 명확한지 확인합니다.
  - 기본 기준: 건물 → 층 높은 순 → 호실 오름차순.
- 오늘 날짜 표시가 실제 오늘 날짜에만 붙는지 확인합니다.
- 1박 2일 예약이 최소 2칸 이상으로 보이는지 확인합니다.
- 체크인 전 미래 날짜가 `투숙 중`으로 표시되지 않는지 확인합니다.
- 체크인, 체크아웃, 취소, 노쇼 상태가 예약 목록과 타임라인에서 일치하는지 확인합니다.
- 단체 예약으로 생성된 객실/투숙객이 단체 관리와 예약 관리에서 동일하게 이어지는지 확인합니다.

### 4.2 단체/행사 관리

- 행사 상태가 서로 배타적인지, 또는 복수 상태를 동시에 가져야 하는지 업무적으로 검토합니다.
  - 예: 지난 행사이면서 정산 필요인 경우 두 조건 모두 필터에 반영되어야 함.
- 업체는 업체 관리 데이터에서 선택되어야 하며, 행사 화면에서 임의 미지정 데이터가 나오면 안 됩니다.
- 행사 담당 영업 직원과 업체 담당자를 혼동하지 않도록 라벨을 확인합니다.
- 객실 배정, 투숙객 등록, 변경 이력이 비어 있지 않은 시연 데이터를 포함합니다.

### 4.3 객실 / 요금 관리

- 객실 유형은 고객이 관리하는 데이터이므로 임의 번역하지 않습니다.
- 객실 유형 정렬은 객실명/구분이 함께 묶이도록 정렬합니다.
- 요금 캘린더와 일괄 수정의 객실 유형 순서가 동일해야 합니다.
- 월 변경, 장기 예약, 미래 요금 등록이 동작해야 합니다.

### 4.4 하우스키핑 / 시설보수

- 하우스키핑 KPI와 필터 카운트가 카드 수와 일치해야 합니다.
- 재실 청소 요청, 청소 시작, 청소 완료 상태가 카드와 필터에서 같은 기준으로 계산되어야 합니다.
- 시설보수는 호실 직접 입력이 아니라 기존 객실 선택이어야 합니다.
- 티켓 발행 일시, 작업 시작 일시, 완료 확인 일시가 구분되어야 합니다.
- `완료` 버튼은 실제 완료인지, `완료 확인`인지 업무 의미를 확인합니다.

### 4.5 매출 분석 / 정산 목록 / 일일 마감

- 매출 분석과 정산 목록은 같은 매출 요약 기준을 사용해야 합니다.
- 정산 기준은 체크아웃입니다.
- 예치금은 매출이 아니라 보관금액으로 분리합니다.
- 미수금, 수납 완료, 완료 대기, 정산 완료 상태가 필터와 명세서에 일치해야 합니다.
- 결제수단별 카드/현금/기타 구분이 정산 흐름에 표시되어야 합니다.
- 일일 마감은 마감 체크와 시재 인수인계를 혼동하지 않아야 합니다.
- 시재는 현금 흐름만 다루고, 정산 검증은 마감 체크에서 다룹니다.

### 4.6 부가서비스

- 통합 POS, 골프, 렌터카는 주문/예약/정산 상태가 일관되어야 합니다.
- 업체와 카테고리는 고객이 관리하는 데이터인지, 시스템 코드인지 구분합니다.
- 완료 전 수정/삭제 가능 여부를 확인합니다.
- 객실 선택 시 고객명이 자동으로 연결되는지 확인합니다.

### 4.7 투숙객 CRM / 멤버십

- 연락처, 이메일, 특이사항은 목록에 과다 노출하지 않고 상세 열람 시 감사 로그를 남깁니다.
- 등급 기준과 등급 변동 이력은 별도 메뉴로 접근 가능해야 합니다.
- 블랙리스트, 수동 등급 변경, 자동 승급 케이스가 데이터에 포함되어야 합니다.
- 고객이 입력한 국적 값은 임의 번역하지 않고 기준 목록 중복을 제거합니다.

### 4.8 직원 / 권한 / 감사 로그

- admin은 한 명만 유지합니다.
- admin 정보는 일반 직원 편집 화면에서 임의 변경하지 않습니다.
- 역할 색상은 중복 선택되지 않아야 합니다.
- 개인정보 열람, 엑셀 다운로드, 예약 변경, 정산 처리, 객실 이동은 감사 로그에 기록되어야 합니다.
- 감사 로그는 기간 필터, 유형 필터, 페이지네이션이 동작해야 합니다.

### 4.9 관리자 화면

- 호텔 대시보드에서 만든 기능이 관리자 화면에도 필요한지 매번 대조합니다.
- 호텔과 관리자 매뉴얼은 분리합니다.
- 관리자 화면의 검색, 정렬, 팝업, 상태 배지는 호텔 화면 기준과 시각적으로 일관되어야 합니다.

## 5. 데이터 적합성 테스트 체크리스트

수정 후 다음 질문에 모두 답해야 합니다.

- 이 값은 사용자가 입력한 데이터인가, 시스템 라벨인가?
- 이 값은 mock API에서 오는가, 화면에 하드코딩되어 있는가?
- 같은 데이터가 다른 화면에서도 같은 금액/상태/일자로 보이는가?
- 오늘 기준 데이터와 미래 데이터가 섞여 오해를 만들지 않는가?
- 완료/미완료/오류/미수금/블랙리스트처럼 예외 케이스가 하나 이상 존재하는가?
- 팝업에서 수정한 값이 목록, KPI, 관련 로그에 반영되는가?
- GitHub Pages에서 localStorage 초기 상태로도 동일하게 재현되는가?

## 7. 완료 기준

작업 완료는 다음 조건을 모두 만족할 때만 말합니다.

1. 관련 파일 수정 완료.
2. 자동 테스트 또는 해당 범위에 맞는 Playwright 직접 검증 완료.
3. GitHub Pages에서 새 커밋 기준 확인 완료.
4. 사용자에게 데이터 기준, 상태 기준, 남은 리스크를 보고.
5. 커밋 및 푸시 완료.

## 8. 사용자 지적사항 누적 규칙

사용자가 화면 오류, 데이터 불일치, 업무 로직 문제, 다국어 혼용, 레이아웃 깨짐을 지적하면 해당 내용은 일회성 수정으로 끝내지 않고 이 문서의 회귀 테스트 케이스에 추가합니다.

운영 규칙:

1. 사용자가 지적한 문제는 `지적 내용`, `원인`, `재발 방지 검증 방법`, `확인 화면`으로 기록합니다.
2. 같은 유형의 문제가 다른 화면에도 있을 가능성을 보고 관련 화면을 함께 검수합니다.
3. 수정 후에는 로컬 테스트뿐 아니라 GitHub Pages 기준으로 재현 여부를 확인합니다.
4. 자동화가 가능한 항목은 Playwright 또는 기존 mock-api 테스트 스크립트에 반영합니다.
5. 자동화가 어려운 항목은 완료 보고 시 수동 검수 결과를 남깁니다.
