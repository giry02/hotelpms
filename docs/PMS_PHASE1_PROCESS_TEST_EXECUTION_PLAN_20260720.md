# Hotel PMS 1단계 프로세스 통합 테스트 실행 계획서

> 문서 버전: 1.0  
> 작성일: 2026-07-20  
> 대상 시스템: Hotel PMS Dashboard + Admin 연계  
> 테스트 기준 영업일: 2026-07-10  
> 현재 상태: **배포본 실행 중 / PASS·PARTIAL 결과는 10장에 기록**

## 1. 문서 목적

이 문서는 화면이 열리거나 버튼을 한 번 클릭할 수 있다는 이유만으로 `PASS`를 주지 않기 위한 실행 기준이다. 각 업무 프로세스를 실제 사용자의 입력, 저장, 재조회, 연계 화면 확인, 예외 처리, 원복까지 수행하고 그 결과를 같은 문서에 기록한다.

기존 테스트 결과에서 다음 조건을 충족하지 않은 `PASS`는 이 문서로 승계하지 않는다.

- 실제 버튼 클릭 대신 DOM 함수 호출 또는 값 주입만 수행한 경우
- 저장 성공 메시지만 확인하고 실제 저장값을 재조회하지 않은 경우
- 페이지 새로고침 후 유지 여부를 확인하지 않은 경우
- 예약 현황, 예약 목록, 타임라인 등 연계 화면의 일치를 확인하지 않은 경우
- 경고창의 취소와 계속 경로를 모두 확인하지 않은 경우
- 콘솔 오류, 페이지 오류, 실패한 네트워크 요청을 확인하지 않은 경우
- 실행 증거가 없는 경우

따라서 이 문서의 모든 결과 초기값은 `NOT TESTED`이다.

## 2. 1단계 범위

1단계는 **업무 프로세스와 상태 전이의 정확성**만 검증한다. 다국어 전체 전수 검증, 모바일 전수 레이아웃, 브라우저별 시각 회귀는 별도 단계에서 수행한다. 단, 1단계 수행 중 업무를 막는 다국어 혼용, 버튼 겹침, 모바일 클릭 불가는 프로세스 실패로 기록한다.

| 영역 | 포함 페이지 | 핵심 검증 |
|---|---|---|
| 인증 | 로그인, 내 정보, 직원 상태 | 계정 상태와 권한에 따른 로그인 허용/차단 |
| 대시보드 | 대시보드 | KPI와 실제 목록 데이터의 일치 |
| 예약 | 예약 현황, 예약 타임라인, 예약 목록 | 등록, 변경, 취소, 체크인, 객실 이동, 체크아웃 |
| 단체 | 단체 업체, 단체 이벤트, 객실 배정, 투숙객 명단, 통계 | 단체 생성부터 투숙·정산까지 연결 |
| 객실 운영 | 객실 관리, 하우스키핑, 시설 보수 | 청소·점검 상태와 예약 가능 여부 동기화 |
| 부가서비스 | 부가서비스 등록, 업체/항목 관리 | 투숙객 기준 노출, 등록, 완료·미완료 전환, 바우처 |
| 정산 | 정산 현황, 정산 목록, 비품 구매, 일일 마감 | 수납, 완료·미완료 전환, 지출·시재 반영 |
| 고객 | 투숙객 관리, 우수 고객 | 고객 저장·수정과 예약/투숙 이력 연결 |
| 직원/권한 | 직원 목록, 권한 설정 | 직원 CRUD, 상태, 역할, 기능 권한 |
| 감사 | 감사 로그, 마감 로그 | 주요 상태 변경의 추적 가능성 |
| Admin 연계 | 호텔/테넌트, 공지, 이벤트 | PMS 데이터와 Admin 노출의 일치 |

## 3. 판정 규칙

### 3.1 결과 상태

| 상태 | 판정 기준 |
|---|---|
| `PASS` | 모든 단계의 화면 결과, 저장값, 새로고침, 연계 화면, 오류 확인, 원복이 모두 성공 |
| `FAIL` | 예상 결과와 실제 결과가 한 항목이라도 다름 |
| `BLOCKED` | 선행 데이터, 권한, 환경 오류로 사용자 경로를 시작하거나 끝낼 수 없음 |
| `PARTIAL` | 일부 단계만 실행했거나 연계 화면·새로고침·원복 검증이 누락됨 |
| `NOT TESTED` | 아직 실행하지 않음 |

### 3.2 PASS 필수 게이트

모든 케이스는 아래 8개 항목을 충족해야 `PASS`로 판정한다.

1. 사용자 화면에서 실제 클릭과 타이핑으로 실행했다.
2. 단계별 예상 화면 결과가 일치했다.
3. API 또는 저장소의 저장 상태가 일치했다.
4. 새로고침 또는 페이지 재진입 후 상태가 유지됐다.
5. 연계 화면의 데이터와 카운트가 일치했다.
6. 콘솔 `error`, `pageerror`, 실패한 API 응답이 없었다.
7. 생성·수정한 테스트 데이터를 원복했다.
8. 스크린샷과 실행 로그가 케이스 ID로 저장됐다.

### 3.3 금지되는 테스트 방식

- `element.value = ...`만 수행하고 `input/change` 이벤트와 실제 저장 버튼을 생략하지 않는다.
- 내부 JavaScript 함수를 직접 호출한 결과를 사용자 버튼 테스트로 인정하지 않는다.
- 버튼 존재, 모달 표시, 토스트 표시만으로 저장 성공을 판정하지 않는다.
- 목록 카운트와 실제 표시 건수가 다르면 어느 한쪽을 임의로 정답으로 간주하지 않는다.
- 테스트가 시간 초과됐다는 이유로 `PASS` 처리하지 않는다.

## 7. 상세 프로세스 테스트 케이스

### 7.1 인증·직원 상태

#### P1-AUTH-001 근무 중 계정 로그인

- **fixture**: `FX-USER-FD`
- **입력**: 이메일 `desk@hotel.example`, 비밀번호 `password123!`
- **절차**
  1. 로그아웃 상태에서 로그인 페이지를 연다.
  2. 이메일과 비밀번호를 실제 입력하고 로그인 버튼을 누른다.
  3. 예약 현황으로 이동한다.
  4. 새로고침 후 로그인 상태를 확인한다.
- **예상 결과**
  - 로그인 후 예약 현황이 열리고 사용자명은 Sarah Connor, 역할은 프론트 데스크로 표시된다.
  - 허용된 메뉴만 표시되고 권한 없는 설정 메뉴는 숨김 또는 접근 차단된다.
  - 새로고침 후 세션이 유지된다.
- **저장/연계 검증**: 로그인 세션의 사용자 ID와 역할이 직원 데이터와 일치한다.
- **정리**: 로그아웃한다.
- **증거**: 로그인 전, 로그인 후 메뉴, 새로고침 후 화면.

#### P1-AUTH-002 휴직·퇴직 계정 로그인 차단

- **fixture**: `FX-USER-LEAVE`, `FX-USER-RETIRED`
- **입력**: `p1.leave@hotel.example` / `P1-Process-2026!`, `p1.retired@hotel.example` / `P1-Process-2026!`
- **절차**: 각 계정으로 로그인 버튼을 누르고 오류 메시지와 세션 생성을 확인한다.
- **예상 결과**
  - 휴직 계정은 `휴직 중인 계정은 로그인할 수 없습니다.` 의미의 안내 후 로그인되지 않는다.
  - 퇴직 계정도 로그인되지 않는다.
  - 대시보드 URL 직접 접근 시 로그인 페이지로 돌아온다.
- **저장/연계 검증**: 로그인 세션 키가 생성되지 않는다.
- **정리**: 관리자 계정으로 재로그인한다.

#### P1-STAFF-001 직원 신규 등록·편집·역할·상태·삭제

- **fixture**: `FX-USER-ADMIN`
- **입력**
  - 이름 `P1 Staff Test`, 사번 `P1-0720`, 이메일 `p1.staff@example.com`
  - 전화 `+63 917 000 7001`, 주소 `Manila Test Address`
  - 입사일 `2026-07-01`, 퇴사일 공란, 역할 `프론트 데스크`, 상태 `근무 중`
  - 비밀번호와 비밀번호 확인 `P1-Process-2026!`
- **절차**
  1. 직원 등록 버튼을 누르고 모든 값을 입력해 저장한다.
  2. 목록에서 생성 행과 모든 표시값을 확인한다.
  3. 편집 버튼을 눌러 전화번호, 주소, 역할을 `총괄 매니저`로 변경해 저장한다.

  5. 상태를 `휴직`으로 변경하고 해당 계정 로그인을 시도한다.
  6. 다시 관리자 계정으로 로그인해 상태를 `퇴직`, 퇴사일 `2026-07-20`으로 변경한다.
  7. 삭제 버튼을 눌러 정책에 따른 확인창과 결과를 확인한다.

## 9. 결함 수정 및 재시험 규칙

1. 결함마다 재현 케이스 ID를 반드시 연결한다.
2. 수정 전 실제값, 원인, 수정 파일, 수정 내용을 기록한다.
3. 수정 후 동일 입력과 동일 fixture로 원 케이스를 처음부터 다시 실행한다.
4. 공통 버튼, 모달, 날짜 입력, 상태 셀렉트를 수정한 경우 관련 전체 케이스를 재실행한다.
5. 수정 코드가 커밋됐더라도 재시험 증거가 없으면 결과는 `PARTIAL`이다.
6. 배포 환경에서 재확인하지 않은 로컬 결과는 `PASS-LOCAL`처럼 별도 기록하고 최종 PASS로 합산하지 않는다.

## 11. 완료 기준

1단계 완료는 다음 조건을 모두 만족할 때만 선언한다.

- 표의 41개 케이스가 모두 실행됐다.
- `NOT TESTED`, `PARTIAL`, `BLOCKED`가 0건이다.
- 치명·높음 결함이 0건이다.
- 실패 수정 후 영향 케이스 재시험이 완료됐다.
- 모든 케이스에 실행 증거가 있다.
- 생성 데이터 원복 결과가 기록됐다.
- 로컬과 배포 환경 결과가 구분되어 있다.
- 최종 커밋 ID와 배포 확인 시각이 결과표에 기록됐다.

이 조건을 만족하지 않으면 테스트가 오래 수행됐더라도 `전체 테스트 완료` 또는 `오류 없음`으로 보고하지 않는다.
