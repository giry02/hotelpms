# Hotel PMS AI 작업 필수요소

이 문서는 새 대화창에서도 동일한 기준으로 작업을 이어가기 위한 필수 작업 규칙입니다. 새 작업을 시작하는 AI는 코드 수정 전에 이 파일과 `docs/USER_ACCEPTANCE_TEST_GUIDE.md`를 먼저 확인해야 합니다.

## 1. 기본 원칙

- 작업 루트는 `E:\AI_Project\Hotel_PMS`입니다.
- 사용자는 GitHub Pages에서 결과를 확인합니다. 로컬 수정만 하고 끝내면 안 됩니다.
- 코드/데이터 수정 후에는 반드시 검증하고, 문제가 없으면 `main` 브랜치에 commit/push까지 완료합니다.
- 관련 없는 untracked 파일은 건드리지 않습니다.
- 사용자가 "답변만", "의견만", "리포트만"이라고 하면 절대 수정하지 않습니다.
- 사용자가 "수정해", "처리해", "가자"라고 하면 구현, 검증, commit/push까지 진행합니다.
- 기존 사용자 변경사항은 되돌리지 않습니다.

## 2. Git 필수 규칙

작업 완료 시 아래 순서를 지킵니다.

1. `git status --short`로 변경 파일 확인
2. 의도한 파일만 `git add`
3. 의미 있는 커밋 메시지로 commit
4. `git push origin main`
5. 최종 답변에 커밋 해시와 검증 결과를 포함

현재 반복적으로 무시해야 하는 untracked 예시는 다음과 같습니다.

- `.codex-remote-attachments/`
- `docs/TEST_REPORT_2026-06-25_INTEGRATION_UAT.md`
- `outputs/i18n-audit/`
- `outputs/pms-requirements-audit-20260615/*.inspect.ndjson`
- `scripts/mock-api/visual-audit-output/`

## 3. 검증 필수 순서

수정 후 최소 검증:

```powershell
node -e "const fs=require('fs'); for (const f of ['dashboard/operations/night-audit.html','dashboard/operations/closing-log.html']) { if (!fs.existsSync(f)) continue; const s=fs.readFileSync(f,'utf8'); const scripts=[...s.matchAll(/<script[^>]*>([\s\S]*?)<\/script>/gi)].map(m=>m[1]).filter(x=>x.trim()&&!x.includes('src=')); scripts.forEach((code,i)=>{ try { new Function(code); } catch(e) { console.error(f+' script '+i+' '+e.message); process.exit(1); } }); console.log(f+' scripts ok'); }"
$env:PMS_BASE_URL='http://127.0.0.1:4177'; npm run smoke:all
```

화면/레이아웃 변경이 있으면 Playwright로 실제 페이지를 열어 다음을 확인합니다.

- 버튼/필터 클릭 전후 레이아웃이 흔들리지 않는지
- 목록 행 높이, 카드 간격, 검색 영역이 다른 화면과 통일되는지
- 모바일/좁은 화면에서 버튼, 텍스트, 팝업이 잘리지 않는지
- localStorage 샘플 초기화 후에도 데이터가 충분히 재생성되는지

반복 지적된 회귀 케이스는 `docs/USER_ACCEPTANCE_TEST_GUIDE.md`에 추가합니다.

## 4. 다국어/데이터 원칙

- 한국어 화면에서는 시스템 라벨, 버튼, 상태, 안내 문구가 한국어여야 합니다.
- 고객이 입력한 데이터는 억지 번역하지 않습니다. 예: 객실 유형명, 업체명, 고객명, 행사명.
- 코드로 정해진 라벨은 다국어 처리 대상입니다. 예: 업체 유형, 상태, 버튼명, 메뉴명.
- 화면에 `undefined`, `??`, 영어/한국어 혼용, 깨진 인코딩이 보이면 반드시 수정합니다.
- 시연 데이터는 실제 업무 흐름을 검증할 수 있을 만큼 다양해야 합니다.

## 5. 호텔 업무 로직 기준

### 예약/체크인

- 입실 예정일 전 예약 상세에서는 체크인 버튼이 보이면 안 됩니다.
- 미래 예약은 예약 취소는 가능하되 체크인은 불가해야 합니다.
- 예약 생성 시 상태를 임의 선택하게 하지 않습니다. 생성 후 체크인/체크아웃/취소 액션으로 상태가 바뀌어야 합니다.
- 체크아웃 전 Folio는 정산 목록에 최종 정산처럼 표시하면 안 됩니다.
- 예치금은 체크아웃 전 받은 금액입니다. 한국어 화면에서는 `예치금`으로 통일합니다.

### 예약현황/하우스키핑

- 예약현황에서 공실도 청소 상태를 확인할 수 있어야 합니다.
- 청소 상태는 `방해금지`, `청소완료`, `청소필요` 중심으로 단순하게 유지합니다.
- 청소 배지는 `청소 완료`, `청소 필요`처럼 청소 상태만 보여야 하며 `재실 중` 같은 투숙 상태와 섞지 않습니다.
- 투숙 중 객실은 새 영업일마다 기본적으로 `청소 필요`로 갱신합니다.
- 빈방은 기존 청소 상태를 유지합니다.
- 단체 예약/블록은 아이콘만이 아니라 단체명도 표시해야 합니다.

### 투숙객/멤버십

- 투숙객 목록에는 이름 정렬과 가입시기 정렬이 필요합니다.
- 개인정보 상세 열람, 엑셀 다운로드는 감사 로그 대상입니다.
- 연락처 같은 개인정보는 목록에 직접 노출하지 않고 상세 열람 기준으로 관리합니다.
- 블랙리스트, 등급 변경 이력, 레이트 체크아웃 등 검증 가능한 샘플 케이스가 있어야 합니다.

### 단체/행사

- 단체/행사는 등록 업체와 연결되어야 하며, 기존 데이터가 업체 없음으로 보이면 시연 불가 데이터입니다.
- 행사 담당 영업 직원과 업체 담당자는 구분해야 합니다.
- 체크인/체크아웃, 총 인원, 행사 유형 등 기본 정보는 라인과 간격을 맞춥니다.
- 변경 이력 데이터가 비어 있으면 시연이 어렵기 때문에 샘플 이력이 필요합니다.

### 부가서비스

- 부가서비스는 객실 기준 등록 흐름을 우선합니다.
- 메뉴는 `부가서비스 등록`, `업체/항목 관리` 중심으로 운영합니다.
- 통합 POS, 골프장, 렌터카는 한 객실 팝업 안에서 선택해 등록하는 흐름이 필요합니다.
- 체크인 예정/공실 객실에는 부가서비스 이용금액이 사전 사용처럼 보이면 안 됩니다.
- 단체의 경우 단체 체크 시 전체 객실이 기본 선택되고 개별 해제가 가능해야 합니다.
- 골프/렌터카 업체별 항목과 바우처 양식은 `업체/항목 관리`에서 관리합니다.
- 객실 카드에는 통합 POS, 골프장, 렌터카, 합산 금액만 요약하고 세부 항목은 상세 팝업에서 봅니다.

### 정산/일일마감

- 모든 정산 기준은 체크아웃입니다.
- 정산 목록에는 개인/단체, 객실/POS/골프/렌터카, 현금/카드/이체, 예치금, 미수금 케이스가 다양하게 있어야 합니다.
- 금일 정산내역은 현금/카드/이체를 구분해서 보여야 합니다.
- 시재 인수인계는 실제 현금 흐름입니다. 카드 매출과 혼동하면 안 됩니다.
- 시재 공식은 `시작 시재 + 금일 현금 정산 + 예치금 - 중간출금 = 기준 현금`입니다.
- 중간출금은 여러 건일 수 있으며 페소/달러/원화를 각각 표시해야 합니다.
- 마감할 때는 다음 근무자 인수인계 메모가 필요합니다.
- 마감 후 다음 담당자는 이전 마감 금액과 인수인계 메모를 확인할 수 있어야 합니다.
- 차액은 사용자가 의도적으로 미수금을 발생시키지 않은 실제 현금 차이로 봐야 합니다.

### 운영 로그/감사 로그

- 운영 로그를 누르면 감사 로그가 먼저 보여야 합니다.
- 감사 로그는 기간 필터가 필요합니다: 금일, 주간, 월간, 기간 선택.
- 개인정보 열람, 다운로드, 예약 변경, 정산 수납, 권한 변경은 감사 로그로 남아야 합니다.
- 마감 로그는 일일마감의 시재/마감 흐름과 같은 기준으로 보여야 합니다.

## 6. 디자인/레이아웃 기준

- 검색 영역, 필터 버튼, 정렬 버튼은 투숙객 관리 기준의 균형을 우선 참고합니다.
- 버튼 클릭, 필터 전환, 페이지네이션 전환 시 레이아웃이 흔들리면 안 됩니다.
- 카드 안에 불필요한 설명을 길게 넣지 않습니다. 운영자가 바로 판단할 정보가 우선입니다.
- 팝업 타이틀과 섹션 타이틀은 작게 흐려지면 안 됩니다.
- 목록 행 클릭으로 상세/관리 팝업이 열려야 하며, 작은 설정 버튼만 의존하지 않습니다.
- 팝업 하단 버튼은 CSS가 적용된 버튼으로 통일하고, 불필요한 닫기 버튼 중복은 줄입니다.
- 데이터가 많은 경우는 페이지네이션, 상세 팝업, 접기/펼치기를 활용합니다.

## 7. 매뉴얼 기준

- 매뉴얼 폴더명은 `manual`입니다. `메뉴얼` 또는 `outputs/manual`로 새로 만들지 않습니다.
- 호텔 사용자용과 어드민용 매뉴얼은 분리합니다.
- 각 화면은 실제 스크린샷 위에 번호 마커를 올리고, 번호별 기능 설명을 제공합니다.
- 팝업은 별도 모음만 만들지 말고, 해당 업무 흐름 안에서 팝업 스크린샷과 설명을 포함합니다.
- 한국어/영어 선택이 가능해야 합니다.
