# Hotel PMS DB·API 연계 후 통합 테스트 계획서

## 1. 문서 목적

이 문서는 현재 Hotel PMS 화면을 실제 데이터베이스와 API에 연결한 뒤 수행할 통합 테스트의 실행 기준이다. 기존 테스트에서 반복적으로 발생한 다음 문제를 재발시키지 않는 것이 목적이다.

- 화면에서는 성공으로 보이지만 저장되지 않거나 새로고침 후 원복되는 문제
- 동일 데이터가 예약 현황, 목록, 타임라인, 객실, 정산에서 다르게 보이는 문제
- 상태 전환 후 버튼·색상·필터 카운트·감사 로그가 이전 상태를 유지하는 문제
- 중복 클릭, 네트워크 재시도, 동시 수정으로 중복 데이터나 금액 불일치가 생기는 문제
- 고정된 테스트 날짜와 로컬 fixture 때문에 실제 업무일에서 로직이 달라지는 문제
- 역할·테넌트·언어·기준 통화가 API와 화면 사이에서 섞이는 문제

이 계획은 페이지가 열리거나 버튼이 눌리는지만 확인하지 않는다. 실제 사용자 조작으로 API를 호출하고, 응답·DB·연계 화면·감사 로그·재접속 결과까지 일치해야 `PASS`로 판정한다.

- 수납, 정산, 지출, 시재, 일일 마감, 통계

- 목록·상세·KPI·필터 카운트·다운로드의 원천 데이터 일치
- 트랜잭션, 참조 무결성, 멱등성, 동시성, 재시도, 장애 복구

- 데스크톱과 태블릿 화면 및 터치 조작
- 응답 시간, 페이지 로딩, 대용량 데이터, 페이지네이션·기간 조회

- 모바일 전용 레이아웃과 768px 미만 화면

외부 시스템은 샌드박스 또는 mock을 사용하되, 우리 API가 외부 실패를 올바르게 처리하는지는 포함한다.

### 4.1 날짜 규칙

고정 날짜 `2026-07-10`을 사용하지 않는다.

| 기호 | 의미 |
|---|---|
| `D0` | API 또는 호텔 설정이 반환한 현재 업무일 |
| `D-1` | D0 전일 |
| `D+1` | D0 익일 |
| `D+30` | D0에서 30일 후 |
| `T0` | 테스트 시작 시각, 호텔 타임존 기준 |

브라우저 시각이 아니라 서버와 호텔의 업무일을 기준으로 한다. 자정 전후 케이스는 별도 QA tenant에서 시간을 제어해 실행한다.

## 6. 판정 규칙

### 6.1 결과 상태

- `PASS`: 아래 12개 게이트를 모두 충족
- `FAIL`: 예상 결과 불일치, 4xx/5xx 오처리, 콘솔 오류, DB 불일치 중 하나라도 발생
- `BLOCKED`: 외부 의존성 또는 환경 장애로 시작할 수 없음
- `NOT TESTED`: 실행하지 않음
- `PARTIAL`: 일부 단계만 실행함. 완료 집계에서는 실패로 취급

### 6.2 PASS 필수 게이트

1. 실제 배포 페이지에서 클릭·터치·타이핑했다.
2. 요청 Method, URI, Query, Header, Body가 API 명세와 일치한다.
3. 응답 Status, Schema, 타입, 길이, enum, null 규칙이 일치한다.
4. DB의 대상 행과 연관 행이 정확히 한 번 변경된다.
5. 실패 케이스에서 부분 저장이 없고 트랜잭션이 rollback 된다.
6. 새로고침과 로그아웃·재로그인 후 동일 상태가 유지된다.
7. 예약 현황·목록·타임라인·객실·정산 등 연계 화면이 동일하다.
8. KPI·필터 카운트·합계가 실제 조회 건수와 일치한다.
9. 허용 역할은 성공하고 비허용 역할과 다른 tenant는 차단된다.
10. 감사 로그의 actor, tenant, 화면명, 동작, 대상 ID, 전후 값, 시각이 일치한다.
11. 한국어·영어에서 서버 코드가 올바른 문구로 표시되고 사용자 입력은 번역되지 않는다.
12. 중복 클릭·재전송·새로고침으로 중복 행이나 중복 금액이 생기지 않는다.

토스트만 확인하거나 API만 직접 호출한 결과는 페이지 통합 테스트의 PASS로 인정하지 않는다.

### 6.3 필수 증거

각 케이스 결과에는 다음 항목을 모두 첨부한다.

| 증거 ID | 내용 |
|---|---|
| `EV-UI-BEFORE` | 시작 화면과 선택 대상 |
| `EV-REQ` | 요청 Method·URI·Header·Body·correlation ID |
| `EV-RES` | 응답 Status·Body·응답 시간 |
| `EV-DB-BEFORE/AFTER` | 대상 및 연관 테이블 전후 조회 |
| `EV-UI-AFTER` | 저장 직후와 연계 화면 |
| `EV-REFRESH` | 새로고침 또는 새 세션 후 화면 |
| `EV-AUDIT` | 감사 로그 행 |
| `EV-CLEANUP` | 테스트 데이터 정리 결과 |

| Page ID | 화면 경로 | 제어 ID/라벨 | API ID·Method·URI | DB 엔티티 | 관련 Case ID | 결과 |

### 9.9 Admin 입점·승인·계정 활성화·PMS 연계

이 영역은 Admin 화면만 별도로 확인하지 않는다. 모든 케이스는 `applicationId → tenantId → tenantAdminUserId → sessionId`를 이어서 기록하고, 공지·광고·지원 문의는 같은 `tenantId`로 대상 여부를 검증한다.

| 업무 단계 | 신청 상태 | Tenant | 최초 관리자 | PMS 접근 | 연계 검증 |
|---|---|---|---|---|---|
| 신청 접수 | `pending` | 없음 | 없음 또는 비활성 credential | 차단 | Admin 신청 목록·KPI·접수 알림 |
| 반려 | `rejected` | 없음 | 없음 | 차단 | 반려 사유·처리자·재신청 revision |
| 승인 | `approved` | `active` | `invited` 또는 활성화 대기 | 활성화 전 차단 | 계약·요금제·기본 설정·승인 알림 |
| 계정 활성화 | `approved` | `active` | `active` | 허용 | tenantId가 포함된 세션·최초 설정 |
| 운영 중 | `approved` | `active` | `active` | 허용 | 대상 공지·광고·지원·과금·감사 로그 |
| 이용 중지 | `approved` | `suspended` | 데이터 보존 | 기존/신규 세션 차단 | 중지 사유·처리자·재활성화 |

`E2E-DBAPI-009`는 이 표의 신청 접수부터 첫 PMS 예약 생성까지를 끊지 않고 수행하는 승인 게이트다. 신청 페이지, Admin 페이지, PMS 페이지를 각각 열어본 결과로 대체할 수 없으며, 하위 단계가 하나라도 실패하면 전체 E2E는 FAIL이다.

## 10. 필수 E2E 시나리오

### E2E-DBAPI-001 개인 고객 전체 사이클

1. `TD-ROOM-CLEAN`에 D0~D+2 예약을 생성한다.
2. 고객과 동반 투숙객을 연결하고 D0에 체크인한다.
3. POS·골프·렌터카·음식점·기타를 각 1건 등록한다.
4. 현금과 카드로 나누어 수납한다.
5. D+1 대신 D0에 조기 체크아웃한다.
6. 정산 완료 후 청소 상태를 확인하고 청소 완료 처리한다.

**최종 기대값**: 예약은 checked_out이며 no-show가 아니다. 객실은 vacant/clean, Folio balance=0, 5개 부가서비스와 결제 이력이 보존되고 고객 이력·감사 로그가 연결된다.

### E2E-DBAPI-002 청소·점검 차단과 복구

1. 청소 필요 객실의 예약 경고에서 취소와 계속을 각각 실행한다.
2. 점검 ticket을 생성하고 예약·체크인을 차단한다.
3. ticket 완료 후 청소 필요를 확인하고 청소 완료로 전환한다.
4. 같은 객실 예약과 체크인을 재시도한다.

**최종 기대값**: 차단 중 저장 0건, 복구 후 1건. 상태 전환 순서와 감사 로그가 일치한다.

### E2E-DBAPI-003 단체 행사 전체 사이클

1. 단체 업체와 행사를 만들고 3실을 배정한다.
2. 대표·동반·미배정 투숙객을 연결한다.
3. 체크인 후 동일 골프·렌터카 서비스와 개별 POS를 등록한다.
4. 객실별 수납과 단체 정산을 완료한다.
5. 통계에서 해당 기간을 조회한다.

**최종 기대값**: 객실·투숙객 없는 배정에는 서비스가 없고, 객실/서비스/총액이 Folio와 통계에서 일치한다.

### E2E-DBAPI-004 지출·시재·일일 마감

1. 현금 PHP 지출, 카드 USD 지출, 현금 KRW 지출을 등록한다.
2. 현금 수납과 환불을 등록한다.
3. 시재 상세를 조회하고 일일 마감한다.
4. 지출 1건을 수정/취소해야 하는 경우 조정 정책을 실행한다.

**최종 기대값**: 통화별 원금이 분리되고 현금만 시재에 반영된다. 마감 합계와 원장이 일치하며 과거 마감이 조용히 변경되지 않는다.

### E2E-DBAPI-005 정산·부가서비스 되돌리기 권한

1. 관리자와 프런트 계정으로 각각 정산 완료/미완료, 부가서비스 완료/미완료를 시도한다.
2. 역할 기능 권한을 변경하고 재로그인 후 반복한다.

**최종 기대값**: 화면과 API 권한이 일치하고 허용된 변경만 저장된다. 모든 성공 전환은 감사 로그에 남는다.

### E2E-DBAPI-006 Admin→PMS 연결

1. Admin에서 tenant 정보·공지·광고·상태를 변경한다.
2. 대상 PMS와 비대상 PMS에서 재조회한다.
3. tenant를 중지·재활성화한다.

**최종 기대값**: 대상 조건, 캐시 무효화, 접근 차단, 재활성화가 일치하며 다른 tenant에 노출되지 않는다.

### E2E-DBAPI-007 네트워크 실패·중복 제출 복구

예약, 부가서비스, 수납, 비품 구매 각각에서 응답 지연→중복 클릭→오프라인→재접속을 실행한다.

**최종 기대값**: 각 업무 데이터는 1건만 존재하고 사용자는 최종 성공/실패 상태를 확인할 수 있다.

### E2E-DBAPI-008 다국어·통화·타임존 왕복

1. TEN-QA-PH와 TEN-QA-VN에서 같은 시나리오를 한국어와 영어로 실행한다.
2. 언어 변경 후 새로고침·메뉴 이동·modal 재오픈을 수행한다.
3. D0 자정 경계에서 예약·지출·감사 시각을 비교한다.

**최종 기대값**: 사용자 입력은 불변, 라벨과 오류만 번역, 기준 통화와 업무일은 tenant 설정을 따른다.

### E2E-DBAPI-009 호텔 가입부터 PMS 첫 업무까지

1. 공개 입점 신청 화면에서 `QA Onboarding Hotel {RUN_ID}`을 신청한다.
2. 승인 전 로그인 차단과 Admin 신청 목록/KPI 반영을 확인한다.
3. Admin이 상세 내용을 검토하고 승인한다.
4. 신청자가 계정을 활성화하고 PMS에 최초 로그인한다.
5. 호텔명·언어·타임존·통화·객실을 설정한 뒤 고객과 예약 1건을 등록한다.

**최종 기대값**: applicationId·tenantId·관리자 userId·sessionId·첫 reservationId가 한 흐름으로 연결된다. 승인 전에는 PMS 데이터가 없고, 승인 후에는 해당 tenant 데이터만 접근 가능하다.

### E2E-DBAPI-010 입점 반려·보완·재신청·승인

1. 필수 정보가 부족한 신청을 Admin에서 구체적 사유로 반려한다.
2. 신청자는 사유를 확인하고 정보를 보완해 재신청한다.
3. Admin은 변경 내용을 비교한 뒤 승인한다.
4. 신청자는 활성화 및 PMS 로그인을 완료한다.

**최종 기대값**: 반려 이력과 revision이 보존되며 활성 신청·tenant·최초 관리자 계정은 각각 1건이다.

### E2E-DBAPI-011 Admin 공지 게시부터 PMS 읽음까지

1. Admin이 한국어/영어 공지를 초안으로 저장한다.
2. 신규 승인 호텔만 대상으로 기간을 설정해 게시한다.
3. 대상/비대상 PMS에서 노출을 비교하고 대상 호텔에서 상세를 읽는다.
4. 게시를 취소하거나 만료시킨 뒤 다시 조회한다.

**최종 기대값**: 대상·언어·기간 조건과 read receipt가 정확하며 비대상 노출과 게시 취소 후 잔존 캐시는 0건이다.

### E2E-DBAPI-012 Admin 광고 계약·게시·노출·중지

1. 광고주/캠페인과 계약 정보를 등록하고 초안 상태를 확인한다.

3. PMS 대시보드에서 광고 상세를 열고 문의를 등록한다.
4. Admin에서 조회·문의·과금을 대조하고 광고를 일시중지한다.

**최종 기대값**: 초안/비대상/기간 외에는 미노출이고, 게시 데이터·상세·문의·과금이 같은 campaignId로 연결된다. 중지 후 빈 상세 팝업은 나타나지 않는다.

### E2E-DBAPI-013 PMS 문의와 Admin 고객지원 왕복

1. PMS 관리자가 문의와 첨부를 등록한다.

3. PMS 관리자가 답변을 확인하고 추가 답변 후 종료한다.

**최종 기대값**: 양쪽의 대화 순서·첨부·상태·알림이 일치하고 전체 이력이 tenant 격리와 감사 로그를 만족한다.

### E2E-DBAPI-014 Tenant 중지·재활성화 운영 통제

2. 기존 API 호출과 새 로그인을 각각 시도한다.
3. Admin이 재활성화하고 다시 로그인해 기존 예약·정산을 조회한다.

**최종 기대값**: 중지 즉시 접근은 차단되지만 데이터는 보존되고, 재활성화 후 동일 tenant 데이터가 복구된다.

| 수정 commit | commit SHA |
| 재시험 결과 | 날짜·실행자·PASS/FAIL |
| 증거 링크 | 캡처·HAR·쿼리·로그 |

## 13. 결함 처리와 재시험

| 등급 | 기준 | 처리 |
|---|---|---|
| P0 | 데이터 유실, 중복 예약/수납, tenant 노출, 로그인·정산 차단 | 즉시 중단·수정·전체 관련 E2E 재시험 |
| P1 | 상태·금액·권한·감사 불일치, 주요 CRUD 실패 | 배포 전 수정·모듈 전체 회귀 |
| P2 | 다국어 혼용, 넘침, 색상·카운트 불일치 | 승인 전 수정·해당 페이지 전수 회귀 |
| P3 | 경미한 문구·정렬 | backlog 가능, 근거 기록 |

수정 후 실패 단계만 다시 눌러 PASS 처리하지 않는다. 해당 케이스를 fixture 초기화부터 끝까지 재실행하고, 같은 상태 엔티티를 사용하는 인접 케이스와 E2E를 함께 재시험한다.
